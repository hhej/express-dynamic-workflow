<objective>
Fix D-04 loop-budget guard so it counts planner iterations within the CURRENT turn only, not the cumulative reasoning_trace. Today's live smoke retest exposed that the cross-turn cumulative check short-circuits turn 2 of a same-thread conversation BEFORE the planner LLM is invoked, causing the response node to re-render turn 1's cached surcharge_result instead of recomputing for the new user message.

Purpose: Restore D-04's documented intent — cap planner *iterations within one user request*, not arbitrary trace length across turns. The reasoning_trace operator.add reducer is intentional (Phase 2 design for the trace UI panel) and stays unchanged; the guard's interpretation of that trace is what changes.

Output: Per-turn windowed budget guard in planner.py + 1 updated unit test + 1 new unit test + 1 E2E test with its workaround monkeypatch removed + ROADMAP backlog 999.4 entry.

    ```python
    def _loop_budget_exhausted(state: dict) -> bool:
        """D-04: True when planner has run >= MAX-1 times in the CURRENT turn.

        999.4 fix (2026-04-25): the reasoning_trace reducer is operator.add
        (Phase 2 design — cumulative trace for the UI panel persists across
        turns), so a length-only check would short-circuit turn 2 of a
        same-thread conversation before the planner LLM is invoked. D-04's
        documented intent is to cap planner *iterations within one user
        request*, so we window the count to "entries since the most recent
        agent='response' entry" (or the entire trace if no response yet).
        """
        trace = state.get("reasoning_trace") or []
        last_response_idx = -1
        for i in range(len(trace) - 1, -1, -1):
            if trace[i].get("agent") == "response":
                last_response_idx = i
                break
        current_turn = trace[last_response_idx + 1:]
        planner_count = sum(
            1 for e in current_turn if e.get("agent") == "planner"
        )
        return planner_count >= PLANNER_MAX_ITERATIONS - 1
    ```

    Note: PLANNER_MAX_ITERATIONS default stays 6. The new check is STRICTER per-iteration (counts only planner-tagged entries) but RESETS per turn.
