## Static audit: _drain_events answer-emit contract

Source: `backend/api/routes/chat.py` lines 138-174.

The `_drain_events` async generator yields exactly one `"answer"`
tuple per `response_node` `on_chain_end` event that carries

```python

    final_payload = output["final_payload"]

    yield "answer", final_payload
```

The graph wiring at `backend/agent/graph.py:273` (`g.add_edge("response", END)`)
means `response` is ALWAYS the final node before graph completion on
any non-paused execution. Combined with the response_node static audit
above (every return carries `final_payload`), this means:

**Variant 1 (graph completes without `response.final_payload` event)**
requires either (a) the graph terminating without ever invoking
response_node, or (b) `_drain_events`'s filter dropping the event.

- (a) is structurally impossible: every conditional-edge path either
  routes to a specialist (which loops back to planner) or to
  `response` (the only terminal edge). The destination-less
  short-circuit added in Plan 03 routes to `respond`, which maps to
  `response` via `_route_from_planner` (graph.py:166), so even the
  short-circuit path lands at response.
- (b) is impossible: the filter condition `name == "response" and
  "final_payload" in output` is satisfied by every response_node
  return per the static audit above.

**Conclusion:** Variant 1 is structurally impossible on the current
codebase.
