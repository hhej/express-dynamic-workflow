# Phase 999.11: Investigate live SSE hang on legit baseline diesel-price query - Context

**Gathered:** 2026-05-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Diagnose AND fix the live `POST /api/chat` hang observed on the legit baseline query `"What's the current diesel price in Bangkok?"` during the [260509-utd](.planning/quick/260509-utd-upgrade-guardrails-to-harden-agent-again/) live end-to-end probe. Symptom: trace stream emits `planner -> fuel_agent -> planner` (3 steps) then no `data: {"type":"answer", ...}` SSE event arrives before the urlopen 60s timeout closes the client; response body is 0 bytes, `status=None`. 4/5 cases in the probe completed cleanly — only this legit baseline hung. Demo-gating for W6.

Three candidate root causes from the backlog entry, to be tested sequentially: (a) SSE termination bug in `_drain_events`/`_fresh_stream`, (b) planner re-loop interacting with the recently-added `tool_call_count` `Annotated[int, operator.add]` reducer, (c) cold-start latency exceeding 60s on a freshly-started uvicorn (Gemini handshake + Google Maps + EPPO CSV read).

Out of scope: redesigning SSE framing, switching event-stream library, or restructuring the agent graph. Out of scope: any change to the locked `calculate_surcharge` formula or the v1.0 grading-gated artifacts.

</domain>

<decisions>
## Implementation Decisions

### Phase scope
- **D-01:** Investigate AND fix in the same phase. Matches prior 999.x debug pattern (999.5/999.6/999.7 all shipped diagnose+fix together). A pure-investigation phase risks needing a follow-up under W6 demo time pressure.
- **D-02:** When the investigation surfaces a fix-vs-mitigation fork, prefer the root-cause fix over a demo-runbook workaround. Stronger demo story; works in any environment without an out-of-band warmup ritual.

### Reproduction protocol
- **D-03:** Primary harness is a fresh-uvicorn + Python httpx script with a 180s client timeout that drains the full SSE stream and prints every event with wall-clock + elapsed-ms timestamps. Living under `.planning/phases/999.11/repro/`.
- **D-04:** Per repro run, capture exactly two artifacts: (1) the full SSE event stream with wall-clock + elapsed-ms timestamps, (2) full uvicorn stderr including any swallowed exceptions inside `_drain_events` / `response_node` / lifespan tasks. Skip Langfuse trace inspection and `/api/conversations` AgentState snapshot for the initial repro pass — pull on-demand only if the stderr+stream pair is insufficient.

### Investigation order
- **D-05:** Test root causes sequentially in this order, NOT in parallel: (c) cold-start latency → (b) planner re-loop / `tool_call_count` reducer → (a) SSE termination bug. Each must be cleanly ruled out before moving to the next so evidence does not get tangled.
- **D-06:** Rule out (c) first because it's cheapest: warm uvicorn with one no-op (or warmup) request, then re-run the legit baseline. If it succeeds under 30s the root cause is cold-start and the fix path becomes lifespan warmup. If it still hangs, (c) is cleanly out and we pivot to (b).
- **D-07:** Test (b) before (a) because the trace evidence (`step 3 = planner`) is a smoking gun for a planner re-loop, and the `tool_call_count` reducer is the newest moving part (added in [260509-utd](.planning/quick/260509-utd-upgrade-guardrails-to-harden-agent-again/) Task 2). Verify deterministically via a focused unit test on `Annotated[int, operator.add]` behavior under parallel fan-out, plus replay against the [guard_input.py](backend/agent/nodes/guard_input.py) cost-bombing branch.
- **D-08:** Test (a) last via static analysis of [chat.py](backend/api/routes/chat.py) `_drain_events` + `_fresh_stream` finally block, then verify against the repro logs.

### Verification standard
- **D-09:** Live bar before declaring 999.11 done: 5 fresh-uvicorn runs of the legit baseline `"What's the current diesel price in Bangkok?"` must each produce an `answer` SSE event within 30 seconds. Restart uvicorn between each run. ~95% demo confidence under cold-start variability.
- **D-10:** Permanent regression: ONE pytest targeting whichever root cause was confirmed. If (b) reducer interaction → unit test asserting `tool_call_count` reducer behavior on parallel fan-out. If (a) SSE → integration test asserting `answer` event arrives on a normal happy path. If (c) cold-start → smoke test asserting lifespan warmup completes deterministically. CI-friendly, no live network.
- **D-11:** Skip the demo-runbook mitigation entry and the Langfuse smoke screenshot. The pytest regression test plus a clean live bar is sufficient evidence; D-02 made root-cause fix the standard.

### Claude's Discretion
- The exact Python httpx script structure, logging format, and how the 5 fresh-uvicorn runs are orchestrated (manual vs. shell loop).
- The granular structure of the regression pytest (file location, fixtures, async vs sync) once the root cause is known.
- How to surface evidence in the eventual SUMMARY.md once the fix lands.
- Whether to add the warmup query as a defensive `lifespan` task even if (c) is ruled out — only if cheap and contained.

<code_context>
## Existing Code Insights

### Reusable Assets
- **httpx async client** — already a dev dependency via langchain stack; the repro script can use `httpx.AsyncClient(timeout=180)` without adding deps.
- **`format_sse` + parse-events helper** — frontend already has `parseSseStream` in `frontend/lib/sse.ts`; the Python repro script can reuse the same `data: <json>\n\n` parser logic (one-liner).
- **`_make_config` + `seed_trace_id`** — when reading Langfuse traces for the failing run, the deterministic 32-hex trace_id pattern `chat_turn_{thread_id}_{turn_idx}` lets us pull the exact run without name lookup.
- **Existing pytest patterns** — `app_with_mocks` fixture pattern in `backend/tests/test_chat_*.py` covers TestClient + lifespan-managed graph + mocked LLM/tools. Regression test will plug into this without scaffolding new fixtures.
- **`EXPRESS_SKIP_COLDSTART_REFRESH=1` env knob** — added by 260509-eum lifespan task. Flipping this in the repro script lets us isolate cold-start fuel refresh from the rest of the lifespan critical path during hypothesis (c).

### Established Patterns
- **`Annotated[int, operator.add]` reducer for parallel fan-out safety** — `tool_call_count` and `errors` and `reasoning_trace` all use this pattern. Hypothesis (b) hinges on whether the reducer pushes count above `MAX_TOOL_CALLS_PER_TURN=6` when both fuel_agent and route_agent return their `+1` increments in the same superstep.
- **Trace status invariants** — `status='ok'`/`'warn'`/`'error'`. The cost-bombing refusal path emits `status='warn'` not `'error'`, which means a silent trip would NOT show as `status='error'` in the SSE — easy to miss.
- **D-13 graceful no-op for Langfuse** — observability calls swallow exceptions so a Langfuse network blip cannot affect user response. But this also means a swallowed Langfuse error during cold start would not surface in the SSE — only in uvicorn stderr (D-04 is calibrated to catch this).
- **D-15.x defense-in-depth pattern** — when adding a fix, prefer one regression test PLUS a static guard (e.g., type signature, exhaustive enum) so the fix cannot silently regress on a future refactor.

### Integration Points
- **Lifespan startup in `backend/api/main.py`** — where 260509-eum schedules background tasks. If hypothesis (c) is confirmed, the fix likely lives here as a `lifespan` warmup task rather than per-request lazy init.
- **`graph.astream_events(version="v2")` in `_drain_events`** — the contract is "one SSE trace event per `on_chain_end` for nodes in `_NODE_NAMES`". Hypothesis (a) might be that `response_node` runs but `final_payload` isn't in the `output` dict (e.g., overwritten by a downstream merger, or the node returns early).
- **`_fresh_stream` finally block** — already enforces "no `done` after `approval_required`" via `pending_approval` flag. Hypothesis (a) candidate: a similar invariant violation where the stream closes silently before `done` even though the graph completed.

</code_context>

<specifics>
## Specific Ideas

- The trace pattern `planner -> fuel_agent -> planner` (step 3 = planner) is the smoking gun for hypothesis (b). User explicitly called this out in the backlog entry: "step 3 was the planner again, suggesting it cycled without producing a response."
- 4/5 cases in the live probe completed cleanly (3 refusals from `guard_input` + 1 false-refusal-via-clarify). Only the legit baseline hung — meaning the bug is on the surcharge/news-query happy path, NOT the refusal path.
- 295/295 backend tests green, but per the backlog entry "deterministic logic only — SSE termination is not unit-tested." This is exactly the gap (a) would exploit.
- User is IT Lead, demo is W6, code freeze is W5. The fix must land cleanly without churn on shipped artifacts.
