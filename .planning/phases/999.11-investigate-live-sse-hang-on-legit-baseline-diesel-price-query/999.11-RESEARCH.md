# Phase 11: Live SSE Hang Root-Cause Fix — Research

**Researched:** 2026-05-11
**Domain:** FastAPI SSE streaming + LangGraph orchestration + cold-start lifespan + reducer-state mechanics under parallel fan-out
**Confidence:** HIGH for code-path analysis (Context7-grade direct reads of the canonical refs); MEDIUM for "which of (a)/(b)/(c) actually trips in the live run" because that is exactly what the locked sequential investigation is supposed to answer empirically

## Summary

The legit-baseline hang occurred on a fresh-uvicorn run. The trace pattern `planner -> fuel_agent -> planner` (3 steps, then 60s timeout, 0-byte body, no `answer` event) is the smoking-gun evidence. Reading the four canonical files line-by-line surfaces:

1. **`fuel_agent_node` is synchronous and contains 3 seconds of blocking `time.sleep` retries** inside `fetch_fuel_price` (Level-1 EPPO scrape is permanently stubbed to raise `NotImplementedError`, but the retry loop still does `time.sleep(0) + time.sleep(1) + time.sleep(2) = 3s` before falling to the CSV). This blocks the asyncio event loop, including the SSE writer task, for 3 seconds per fuel_agent invocation. (HIGH confidence — direct read of `backend/agent/tools/fetch_fuel_price.py:43-54`.)
2. **The `tool_call_count` reducer is `operator.add` with NO clamp** and the cap check (`guard_input`) only fires at graph START, not mid-turn. Inside a single turn the cap is structurally unreachable as a refusal trip — the planner-loop budget (`PLANNER_MAX_ITERATIONS=6`) gates first. So hypothesis (b)'s "silent cost-bomb trip" within one turn is implausible; (b)'s plausible mechanism is "reducer-driven planner re-loop until budget exhausts and runs to respond". (HIGH confidence on mechanism; MEDIUM on whether this is the actual trip.)
3. **`_drain_events` and `_fresh_stream` have one provably silent close-without-answer pattern**: if `astream_events` exhausts AND no `response`-node `on_chain_end` event ever fires with `final_payload`, AND `snapshot.next` is empty (graph claims completion), AND no exception was raised — then the `finally` block emits a bare `done` event with no preceding `answer`. The locked reading of the live log says "no `answer` AND no `done`" — meaning the client timed out BEFORE `astream_events` exhausted, i.e. the run is still in-flight. That points to (b) or (c), not (a). (HIGH confidence on close semantics; HIGH on disambiguation rule.)

**Primary recommendation:** Execute the locked c→b→a investigation order. The repro harness should be a single Python `httpx` script under `.planning/phases/999.11/repro/`; the 5 fresh-uvicorn runs (D-09) should be a shell loop wrapping uvicorn start/stop + the script per run; the regression test should slot into the existing `app_with_mocks` (for (a)) / direct-node-invocation (for (b)) / `app_with_stub_refresh` (for (c)) patterns the suite already uses.

### Pattern 1: Subprocess uvicorn for fresh-process repro (D-03)

**What:** Spawn `uvicorn backend.api.main:app --port 8765` as a subprocess from the repro script, wait for readiness via a `/health` poll, then issue the legit-baseline POST.

**When to use:** This is the only way to deterministically capture cold-start latency without relying on a developer to remember `Ctrl-C` and `uvicorn ...` between runs.

**Example:**

```python
# Source: pattern derived from backend/tests/test_coldstart_fuel_refresh.py
#         (which uses TestClient instead; we need a real subprocess).
import asyncio
import json
import os
import signal
import subprocess
import time
from datetime import datetime, timezone
from pathlib import Path

import httpx

REPO_ROOT = Path(__file__).resolve().parents[3]


def _wall_clock() -> str:
    return datetime.now(timezone.utc).isoformat().replace("+00:00", "Z")


async def probe_once(stderr_path: Path, sse_path: Path) -> dict:
    """Run a single fresh-uvicorn probe.

    Returns dict with: started_at, first_event_at_ms, answer_at_ms (or None),
    done_at_ms (or None), elapsed_total_ms, exit_code, event_count.
    """
    stderr_f = open(stderr_path, "wb")
    proc = subprocess.Popen(
        ["uvicorn", "backend.api.main:app", "--port", "8765", "--log-level", "info"],
        cwd=str(REPO_ROOT),
        stderr=stderr_f,
        stdout=subprocess.DEVNULL,
    )
    try:
        # Readiness gate: poll /health up to 30s. This counts toward
        # the cold-start budget — the user's 30s D-09 window starts
        # when /api/chat is HIT, not when uvicorn binds.
        start_ts = time.monotonic()
        async with httpx.AsyncClient(timeout=2) as client:
            for _ in range(60):  # 30s @ 0.5s
                try:
                    r = await client.get("http://127.0.0.1:8765/health")
                    if r.status_code == 200 and r.json().get("graph_ready"):
                        break
                except Exception:
                    pass
                await asyncio.sleep(0.5)
            else:
                raise RuntimeError("uvicorn did not become healthy in 30s")

        # The actual probe.
        request_start = time.monotonic()
        events = []
        async with httpx.AsyncClient(timeout=180) as client:
            async with client.stream(
                "POST",
                "http://127.0.0.1:8765/api/chat",
                json={"message": "What's the current diesel price in Bangkok?"},
            ) as r:
                async for line in r.aiter_lines():
                    if not line.startswith("data: "):
                        continue
                    elapsed_ms = int((time.monotonic() - request_start) * 1000)
                    env = json.loads(line[len("data: "):])
                    events.append({
                        "wall_clock": _wall_clock(),
                        "elapsed_ms": elapsed_ms,
                        "type": env["type"],
                        "payload": env["payload"],
                    })

        sse_path.write_text(json.dumps(events, indent=2, ensure_ascii=False))
        return {
            "uvicorn_ready_ms": int((request_start - start_ts) * 1000),
            "event_count": len(events),
            "types": [e["type"] for e in events],
            "first_answer_ms": next(
                (e["elapsed_ms"] for e in events if e["type"] == "answer"), None
            ),
            "done_ms": next(
                (e["elapsed_ms"] for e in events if e["type"] == "done"), None
            ),
            "total_elapsed_ms": (
                events[-1]["elapsed_ms"] if events else None
            ),
        }
    finally:
        proc.send_signal(signal.SIGINT)
        try:
            proc.wait(timeout=10)
        except subprocess.TimeoutExpired:
            proc.kill()
        stderr_f.close()
```

## Hypothesis (c): Cold-Start Latency — Sequential Step 1 per D-05/D-06

### What "cold-start" actually means in this codebase

Source: `/Users/pollot/Desktop/express-dynamic-workflow/backend/api/main.py` (lines 70-101).

The lifespan does five things, in order:

1. `aiosqlite.connect(CHECKPOINT_PATH)` — opens the SQLite checkpointer (fast; local disk).
2. `await checkpointer.setup()` — idempotent table creation (fast).
3. `build_graph(checkpointer)` — compiles the LangGraph StateGraph at import time of the node modules. Triggers Python imports of `langchain_google_genai`, `googlemaps`, `langfuse`, etc. Each first-time import takes ~100-500ms.
4. **Schedules** `_coldstart_fuel_refresh()` as `asyncio.create_task(...)` (Quick 260509-eum). NOTE: scheduled BEFORE `yield`, BUT NOT awaited — so the task runs concurrently with request handling.
5. `yield` — lifespan unblocks; API begins accepting traffic.

**The critical realization:** the slow path on the first request is NOT the lifespan. The lifespan completes once the imports + `aiosqlite.connect` resolve, typically < 2s. The slow path is everything that happens INSIDE the first `/api/chat` request:

| Per-request cold-start cost (first turn only) | Where it lives | Time estimate |
|-----------------------------------------------|----------------|---------------|
| `get_chat_model()` instantiating `ChatGoogleGenerativeAI` | `backend/agent/llm.py:16-30` — called inside planner_node + fuel_agent_node + route_agent_node + pricing_agent_node + search_agent_node | First call ~500ms-2s (TLS handshake + model client init); subsequent calls cheap |
| First Gemini API call | `model.invoke(...)` inside `planner_node` | ~1-3s typical (network + LLM latency) |

## Hypothesis (b): `tool_call_count` Reducer + Parallel Fan-Out — Sequential Step 2 per D-05/D-07

### Reducer mechanics — exact ground truth

Source: `backend/agent/state.py:112-129`:

```python
tool_call_count: Annotated[int, operator.add]
```

LangGraph applies `operator.add` to merge concurrent writes from nodes that run in the same Pregel superstep. Behavior:

- A node returns `{"tool_call_count": 1}` → reducer applies `current + 1`.
- Two nodes return `{"tool_call_count": 1}` in the same superstep → reducer applies `current + 1 + 1 = current + 2`.
- A node returns `{"tool_call_count": -3}` (the guard_input reset path) → reducer applies `current + (-3) = current - 3`.

There is NO clamp, NO ceiling, NO floor.

### Where increments and resets happen — exact ground truth

| Site | Code | Behavior |
|------|------|----------|
| `fuel_agent_node` return | `backend/agent/nodes/fuel_agent.py:142` | `"tool_call_count": 1` |
| `route_agent_node` return | (analogous, per `test_tool_call_counter.py::test_route_agent_increments_count`) | `"tool_call_count": 1` |
| `search_agent_node` return | (analogous) | `"tool_call_count": 1` |
| `pricing_agent_node` return | (analogous) | `"tool_call_count": 1` |
| `planner_node` return | `backend/agent/nodes/planner.py` (verified by `test_planner_does_not_increment`) | **Never touches `tool_call_count`** — planner is a router, not a tool-caller |
| `guard_input_node` reset | `backend/agent/nodes/guard_input.py:218-221` | On fresh turn (user_count > response_count): `reset_delta = -tool_count` |
| `guard_input_node` cap trip | `backend/agent/nodes/guard_input.py:231-262` | When `not fresh AND tool_count >= 6`: emits `next_step='respond'` + `category='cost_bombing'` + status='warn' |

### Why "silent cost-bomb trip within one turn" is structurally implausible

The cap check fires INSIDE `guard_input_node`. The graph wiring (graph.py:234-242) is:

```
START -> guard_input -> {planner | response}
```

After the planner routes to a specialist, the specialist returns to planner via `g.add_edge("fuel_agent", "planner")` (graph.py:255). **There is no edge back to `guard_input` mid-turn.** So `guard_input_node` runs EXACTLY ONCE per turn, at the start, before any tool call. The cap check sees `tool_count` BEFORE any increment from this turn — meaning the only way `tool_count >= 6` at guard entry is if a PRIOR turn left it that high without resetting (i.e. `_is_fresh_turn` returned false because user_count == response_count).

That cannot happen in our setup: the first user message on a fresh thread has user_count=1, response_count=0 → fresh → reset to 0 → no cap trip. Subsequent turns ALSO reset (since `_is_fresh_turn` triggers on user_count > response_count, which is true on every new POST).

**Conclusion: "silent cost-bomb trip" is NOT how (b) would manifest on the legit-baseline first-turn run.** Demoting hypothesis (b)'s likelihood, but it remains plausible via a different mechanism (see next).

## Common Pitfalls

### Pitfall 1: Confusing "stream closed silently" with "stream still in-flight at client timeout"

**What goes wrong:** Reading "no answer SSE event arrives before urlopen 60s timeout" as "the server emitted everything and forgot to send `done`" — when the actual symptom is "the server was still in the middle of generating the response."

**Why it happens:** The original backlog phrasing leans toward (a), but the trace evidence "step 3 = planner" strongly suggests an in-flight node, not a stream-close bug.

**How to avoid:** The repro script MUST distinguish "stream closed (HTTP body terminated)" from "stream still open (client timed out)". httpx's `r.aclose()` semantics make this explicit. Capture both `stream-close timestamp` (if any) AND `client-timeout timestamp`.

**Warning signs:** If you find yourself proposing a `_drain_events` patch BEFORE running the repro, you've assumed (a) without evidence.

### Pitfall 2: Tagging a non-planner trace entry with `agent='planner'`

**What goes wrong:** `_loop_budget_exhausted` counts entries by `entry.agent == 'planner'`. An over-tag causes premature budget exhaustion, which routes to `respond` with `clarification_reason='planner_loop_budget_exhausted'` — and response_node renders a generic clarify, not an answer.

**Why it happens:** Easy to default to `agent='planner'` when emitting a trace entry from any branch of `planner_node`.

**How to avoid:** Audit every trace-entry emission in `planner.py`. Currently only ONE branch (the search_context short-circuit, line 248-258) emits a planner trace explicitly; the parse_failed and budget-exhausted branches emit NO trace. This is correct.

**Warning signs:** If the regression test fix involves adding a trace entry tagged `agent='planner'`, double-check that it should be `agent='response'` or `agent='guard_input'` instead.

### Pitfall 3: Repro using TestClient instead of real uvicorn

**What goes wrong:** TestClient runs in-process with a single asyncio loop and does NOT exercise real network. Cold-start latency from Gemini handshakes, Langfuse TLS, etc. would not reproduce. The bug might not manifest.

**Why it happens:** TestClient is the convenient test framework, and the codebase uses it heavily.

**How to avoid:** D-03 LOCKS the use of a fresh-uvicorn subprocess. The Pattern 1 example above shows the correct shape. Use TestClient only for the regression pytest, not for the repro harness.

**Warning signs:** If your repro script imports `from fastapi.testclient import TestClient`, you're doing it wrong.

### Pitfall 4: Adding a defensive timeout that masks the root cause

**What goes wrong:** Wrapping `_drain_events` in `asyncio.wait_for(..., timeout=N)` so the SSE handler emits `error + done` if the graph hangs — solves the symptom (no more 60s timeout, client gets a clear error) but hides the underlying node-level hang.

**Why it happens:** It's a tempting defense-in-depth pattern.

**How to avoid:** Deferred Ideas explicitly excludes this: "Per-request soft timeout that emits `error` + `done` if any node exceeds N seconds. Defensive feature; design separately." Stay out of scope.

**Warning signs:** If a plan task mentions `asyncio.wait_for` around the graph invocation, push back.

### Pitfall 5: Polluting `data/checkpoints.db` across repro runs

**What goes wrong:** Each repro run uses a random thread_id (uuid4). Old checkpoints accumulate. If a prior run left `tool_call_count` at a high value AND the repro reuses the same thread_id (e.g., for resume testing), `guard_input` may trip cost-bombing.

**Why it happens:** Forgetting to clear state between runs.

**How to avoid:** The repro script generates a FRESH `thread_id = str(uuid4())` per probe. The 5 runs are 5 DIFFERENT thread_ids. After all 5 runs, optionally delete `data/checkpoints.db` (or just trust the random thread_ids to isolate).

**Warning signs:** Failure rate varies between probe runs in a way that depends on test order.

### Pitfall 6: Regression test relies on the live network

**What goes wrong:** The regression test calls real Gemini or real Tavily, fails in CI, gets disabled.

**Why it happens:** The fix wants "the legit query end-to-end" and the easy way is to call real services.

**How to avoid:** D-10 explicitly requires "CI-friendly, no live network." The `app_with_mocks` fixture in `test_api_chat.py` shows the pattern: monkeypatch `get_chat_model` to return `FakeMessagesListChatModel`; monkeypatch `fetch_fuel_price` to return a static `FuelData`.

**Warning signs:** If the test fixture does NOT call `monkeypatch.setattr(fuel_mod, "fetch_fuel_price", ...)`, it will try to read the real CSV and possibly hit the (stubbed) network — and even with the CSV fallback it will burn the 3-second blocking sleep, slowing CI.

### Pitfall 7: Bumping `MAX_TOOL_CALLS_PER_TURN` or `PLANNER_MAX_ITERATIONS` as the "fix"

**What goes wrong:** Increasing the budget makes the hang less likely but doesn't remove the cause.

**Why it happens:** It's a simple env-var change.

**How to avoid:** Resist; root-cause D-02 wins. If the planner re-loops, fix the loop logic, not the budget.

**Warning signs:** A plan that proposes only an env-var bump.

   - What we know: the keys are read from `.env` at module import time; absent keys → graceful no-op.
