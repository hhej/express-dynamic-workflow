# Phase 999.11 Plan 03: Hypothesis (b) Planner Re-Loop — CONFIRMED + FIXED

**Live SSE hang on legit baseline diesel-price query CLOSED. Cache-aware override cascade in planner.py was unconditionally mis-routing destination-less follow-ups to fetch_route after fuel was cached, causing route_agent to ValueError on the missing destination, the SSE stream to emit `done` with errors[], and the client to time out at 60s with a 0-byte body. Surgical pre-LLM short-circuit closes the gap; 5/5 fresh-uvicorn runs PASS_UNDER_30S at ~7.6-7.9s each.**

## Decisions Made

- **CONFIRMED path executed (not RULED OUT path).** Task 2 ran; Task 3 was correctly skipped. EVIDENCE.md verdict line at line 7 reads `Hypothesis (b) planner re-loop: CONFIRMED` (exact grep target).

- **Short-circuit placement: BEFORE the LLM invoke** (not after). Rationale: (a) avoids wasting a Gemini call when the next step is structurally determined; (b) prevents the cache-aware override at planner.py:500-509 from firing on the LLM's emission — the LLM still emits `fetch_fuel + surcharge_query` on the destination-less baseline question (verified in EVIDENCE.md row 4), and the override would re-trigger the bug if the short-circuit ran post-LLM.

- **Four pre-conditions (all must hold to fire):** `state.fuel_data is not None`, `not state.destination`, `not state.shipping_type`, `not state.weight_kg`. The four together describe the legit-baseline shape exactly (a follow-up after fuel was fetched with no logistics fields ever extracted). A real surcharge follow-up always carries >= 1 prior logistics field from the previous turn's extraction (the planner's 999.1 merge surfaces it as `merged_shipping_type`/etc.), so the short-circuit never fires on the surcharge path.

- **Defaulting user_intent to 'news_query' on short-circuit** when `state.user_intent` is None. The existing response_node logic renders the clarify markdown when no logistics fields are present, which is the correct UX for a destination-less fuel-price query. (Future enhancement out-of-scope: render the cached fuel price prose explicitly — this plan only addresses the HANG.)

- **Rule 1 deviation: 3 pre-existing test fixups.** `test_skips_fetch_when_fuel_fresh` / `test_planner_no_fanout_when_fuel_fresh` / `test_trace_tool_output_reflects_post_override_next_step` were testing a non-production-realistic synthetic state shape. Updated state with `shipping_type='bounce'` pre-population. Docstrings note the FIX-02 rationale inline.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] 3 pre-existing planner tests had non-production-realistic state shape**

- **Found during:** Task 2 Step B (GREEN fix landing) — `pytest tests/test_planner.py` regressed 3 tests after the short-circuit was added.
- **Issue:** `test_skips_fetch_when_fuel_fresh`, `test_planner_no_fanout_when_fuel_fresh`, and `test_trace_tool_output_reflects_post_override_next_step` were constructing synthetic states with `fuel_data` populated but no other state fields set, expecting the cache-aware override + fetch_route routing to fire. In real production, a state with cached fuel always inherits prior logistics fields from the previous turn's extraction (the 999.1 merge propagates them), so the synthetic shape was non-production-realistic. The destination-less short-circuit correctly catches the synthetic shape (it IS the bug shape!) but breaks these tests.
- **Fix:** Added `shipping_type='bounce'` pre-population to each test's state, making them representative of the realistic 'follow-up with cached fuel + prior shipping_type' pattern that the cache-aware override + fetch_route routing is supposed to exercise. Docstring on each test updated with the FIX-02 rationale.
- **Files modified:** `backend/tests/test_planner.py` (3 test functions, ~6 lines added each: 1 line for `shipping_type='bounce'` + ~5 lines for docstring update).
- **Commit:** `e550256` (alongside the GREEN fix).
