# Phase 999.11 — Live SSE Hang Root-Cause Fix — SUMMARY

**Phase:** 999.11 (Phase 11 / v1.1)

## Outcome

The live `POST /api/chat` hang on the legit baseline diesel-price query (`"What's the current diesel price in Bangkok?"`) was caused by `planner_node`'s cache-aware override cascade (`backend/agent/nodes/planner.py:500-509`) unconditionally promoting `next_step` to `fetch_route` on destination-less follow-ups once fuel was cached — `route_agent_node` then raised `ValueError("requires 'destination'")` and the SSE stream emitted `error + done`, leaving the client with no `answer` event. Hypothesis (b) **CONFIRMED + FIXED** by a surgical pre-LLM destination-less short-circuit in `planner_node` (commit `e550256`, Plan 03) gated on four pre-conditions (`fuel_data` populated AND no destination AND no shipping_type AND no weight_kg). Hypotheses (c) cold-start and (a) SSE termination were cleanly **RULED OUT** in Plans 02 and 04. Post-fix live bar: **5/5 fresh-uvicorn runs PASS_UNDER_30S** at 7.6-7.9s each (D-09 demo gate cleared). Backend pytest: 358/358 GREEN (+3 over the 355 pre-phase baseline; no regressions).

## Root cause

Hypothesis (b) — planner re-loop via cache-aware override mis-firing on destination-less follow-up.

**File:line:** `backend/agent/nodes/planner.py:500-509` (cache-aware override cascade) combined with `backend/agent/nodes/planner.py:99-126` (`_route_matches` early-returns False on `not destination`).

**Deterministic failure mechanism (verbatim from Plan 03 EVIDENCE.md "Conclusion"):**

1. Turn 1: user asks `"What's the current diesel price in Bangkok?"`.
2. `guard_input` allows (domain match on "diesel"); no trace entry on the allow path.
3. `planner` pass 1: LLM emits `next_step=fetch_fuel`, `destination=None`, `user_intent=surcharge_query` (the question reads as fuel-related; the LLM picks the closest schema `next_step`).
4. `fuel_agent` runs successfully, populates `state.fuel_data` with a `fetched_at` timestamp at the current time, emits +1 to `tool_call_count`.
5. `planner` pass 2: LLM again emits `next_step=fetch_fuel` (same question, same prompt — LLM is stateless from its own perspective).
6. **Cache-aware override fires** at `planner.py:500`: `_fuel_fresh(state) == True` → `_route_matches(state, None, None)` → False (destination is None) → `else: next_step = "fetch_route"` at `planner.py:509`.
7. `route_agent_node` runs and raises `ValueError("route_agent_node requires 'destination' in state; got destination=None")` per Plan 05-09 D-10 (selective ValueError catch on missing origin/destination).
8. The graph's `_wrap_error_sink` (per Phase 03 D-24) re-raises `ValueError` for `route_agent`; the SSE handler emits an `error` event from the broad-`except` block at `chat.py:289`, then `done` from the `finally` block. Total elapsed ~10s. The client receives `meta + trace × 3 + error + done` with no `answer` event — observed in the field as the 60s urlopen timeout (the original 260509-utd live probe), in this reproduction environment as a clean ~10s `error + done` close.

The bug never fires on a real surcharge follow-up because realistic state always carries >= 1 prior logistics field from the previous turn's extraction (the 999.1 merge surfaces `merged_shipping_type` / etc.).
