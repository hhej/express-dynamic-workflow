## Rate table re-architecture

### Q7: How should the rate table change to incorporate origin?

| Option | Description | Selected |
|--------|-------------|----------|
| Origin-zone × destination-zone matrix | rate_table grows to 135 rows; SQL stays simple | ✓ |
| Keep dest-only base + origin distance multiplier | No schema change; pricing logic moves into Python | |
| Drop rate table; price purely from Google Maps distance | base = distance_km × THB_per_km; loses table-as-source-of-truth | |

**User's choice:** Origin-zone × destination-zone matrix (recommended). Locks D-05.

### Q8: Pricing philosophy for the matrix?

| Option | Description | Selected |
|--------|-------------|----------|
| Symmetric distance-style — same-zone cheapest, cross-zone scales | Diagonal cheapest; M[i,j] = M[j,i] | ✓ |
| Asymmetric — shipping FROM dense zones costs more | Realistic labor/rent economics | |
| Simple zone-distance kicker on top of current dest-zone base | Two factors combined (dest-difficulty + zone-distance penalty) | |

**User's choice:** Symmetric distance-style (recommended). Locks D-06.
