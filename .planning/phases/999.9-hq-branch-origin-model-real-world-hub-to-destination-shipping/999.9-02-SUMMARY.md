## Decisions Made

### API-boundary default behaviour (Pitfall 1)

The API boundary (chat handler) defaults `origin_hub_id` to `'hq-lat-krabang'` BEFORE seeding `initial_state`:

```
ChatRequest.origin_hub_id  →  _fresh_stream  →  resolved_hub_id  →  initial_state["origin_hub_id"]
        |                                          |
        ├── None              →  'hq-lat-krabang' (default)
        ├── invalid value     →  'hq-lat-krabang' (allowlist mismatch)
        └── valid hub_id      →  passthrough
```

**Consequence for the D-09 narration bullet:**

| Path                                                         | state.origin_hub_id at pricing_agent entry | D-09 bullet fires? |
| ------------------------------------------------------------ | ------------------------------------------ | ------------------ |
| API integration (POST /api/chat with no origin_hub_id)      | `'hq-lat-krabang'` (boundary default)     | NO                 |
| API integration (POST /api/chat with valid origin_hub_id)   | the supplied value                         | NO                 |
| API integration (POST /api/chat with invalid origin_hub_id) | `'hq-lat-krabang'` (boundary default)     | NO                 |
| Direct unit call to `pricing_agent_node` with `state.origin_hub_id=None` | `None` | YES (D-09 unit-test path) |

The D-09 narration bullet is exercised by `test_pricing_agent_default_to_hq_when_none` in `test_pricing_agent.py`, NOT at the API integration layer (`test_chat_request_no_hub_default_narration` asserts the bullet is suppressed there).

## Deviations from Plan

### Rule 3 deviation — `_route_matches` updated for Phase 999.9 cache lookup

**Found during:** Task 1 verification (`test_followup_only_runs_pricing` failed with `route_calls[n]==2 instead of 1`)

**Issue:** `_route_matches` in `planner.py` compared `route_data["origin"]` against the planner's free-text `merged_origin`. Wave 1 + Task 1 introduced two changes that broke this:

1. After Task 1, `RouteData.origin` is the resolved hub address (e.g., "Lat Krabang Industrial Estate, Bangkok"), not the raw `origin` arg.
2. Planner's `merged_origin` comes from prose extraction (e.g., "Bangkok").

These two strings never match → cache lookup always misses → follow-up turns spuriously re-fetch the route.

**Fix:** `_route_matches` now compares on `origin_hub_id` when both sides carry it; falls back to legacy free-text origin compare only for pre-999.9 RouteData payloads.

**Files modified:** `backend/agent/nodes/planner.py:_route_matches`.

**Why in scope of Task 1 (not a separate plan):** This was a direct side-effect of Task 1's signature change to `calculate_route`. Without it, the test suite Wave 1 verification gate would not pass.

### Rule 3 follow-on — `test_graph.py` mock fixtures wired for hub_id

**Found during:** Task 1 full-backend pytest run (`test_graph.py` had 1 failure on `test_followup_only_runs_pricing` cascading from the `_route_matches` fix above, plus implicit assumptions in the count_route mocks).

**Issue:** Three `count_route` helpers in `test_graph.py` returned `RouteData` without `origin_hub_id`, so `_route_matches` (after the above fix) fell through to legacy compare and missed the cache. Three static `RouteData(origin="Bangkok", ...)` returns also needed `origin_hub_id` for symmetry. `_empty_state` needed seeding with `origin_hub_id="hq-lat-krabang"` to mirror what the API boundary (Task 3) seeds in production.

**Fix:** Added `origin_hub_id` to all three `count_route` helpers + four static `RouteData(...)` returns + `_empty_state` seed (with explanatory comment about Pitfall 1).

**Files modified:** `backend/tests/test_graph.py` (4 sites total).

**Why in scope of Task 1:** Direct side-effect of Task 1's signature change. The test_graph file was already in the modified-files list per the plan's frontmatter.

### Token-detection refinement for follow-up hub inheritance (in-task adjustment)

**Found during:** Task 2 test execution (`test_planner_followup_explicit_hub_in_prose_wins` initial implementation failed)

**Issue:** Initial token-detection clause only checked the FIRST comma-split chunk of each hub address (`"Mueang Nonthaburi"` for `branch-nonthaburi`). User messages like "What about from Nonthaburi?" use the bare province name — so the check missed valid prose mentions.

**Fix:** Expanded token list to include both comma-split chunks AND bare provinces (stripping "Mueang " prefix). E.g., for `branch-nonthaburi` we now check `["mueang nonthaburi, nonthaburi"]` becomes `["mueang nonthaburi", "nonthaburi"]` PLUS `["nonthaburi"]` from the prefix-strip.

**Files modified:** `backend/agent/nodes/planner.py` (token-detection block in followup branch).

**Why in scope of Task 2:** Same task, same code block — this was inline iteration to match plan behavior, not a deviation from plan intent.
