# Phase 999.9: HQ/Branch Origin Model - Research

**Researched:** 2026-05-10
**Domain:** Multi-agent state extension + SQLite schema migration + Next.js 15 / React 19 selector UX
**Confidence:** HIGH (most decisions already locked in CONTEXT.md; remaining gray areas resolved against existing project conventions)

## Summary

Phase 999.9 adds an explicit origin hub (1 HQ + 9 branches) to the existing destination-only zone pricing model. CONTEXT.md D-01..D-10 lock the architecture: the 10-hub list, the 135-row origin_zone × dest_zone × ship_type × weight_tier rate matrix, the symmetric multiplier shape, single-leg routing, hybrid dropdown+prose origin capture, and silent default-to-HQ when unspecified. Research therefore focuses on the gray areas: numeric multiplier values, schema migration mechanics, cache key extension, planner prompt token budget, and frontend HubPicker patterns.

The codebase has every reusable seam already in place. `seed_database.py` uses `pandas.DataFrame.to_sql(..., if_exists="replace")` — schema migration is *free* (drop+recreate), no ALTER TABLE needed. `AgentState` already follows additive-only TypedDict extension (Phase 3 D-05, Phase 5 D-07/D-11 precedent). `TTLCache` keys are arbitrary tuples — extending `(origin, destination)` to `(origin_hub_id, destination)` is a one-line change. The Planner extraction call already returns Title-Case `origin` strings; adding a sibling `origin_hub_id` field to PlannerOutput is a 5-line schema change plus an allowlist validator. Phase 4-era `ChatColumn` accepts an arbitrary `placeholder` prop and a Phase 5/6 props chain for HITL — `HubPicker` plugs in next to `ChatInput` with the same prop pattern.

**Primary recommendation:** Treat 999.9 as a "shape-additive" phase across 3 layers. (1) Data: NEW `data/raw/hubs.json` mirroring `zone_definitions.json`; rate CSV gains an `origin_zone` column; rate_table SQLite re-seeded via `if_exists="replace"`. (2) Backend: AgentState + RouteData + ChatRequest get an additive `origin_hub_id: Optional[str]`; lookup_rate signature extends to `lookup_rate(shipping_type, origin_zone, dest_zone, weight_kg)`; route cache key extends to `(origin_hub_id, destination)`; Planner prompt gets a 10-line hub shortlist injection. (3) Frontend: NEW `HubPicker.tsx` selector + a `Hub` type mirror; ChatRequest body gains optional `origin_hub_id`. **Critical invariant:** every existing Phase 1-8 contract must remain runnable on a request that omits `origin_hub_id` — the planner falls back to dropdown's value (default HQ Lat Krabang on cold start), so absence never crashes the agent.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Hub data model & seed (D-01 through D-04)**
- **D-01:** Hub object: `{hub_id, name, address, zone}`. Seeded from `data/raw/hubs.json` mirroring `data/raw/zone_definitions.json` shape (top-level keyed object with named child fields).
- **D-02:** 10 hubs total: 1 HQ + 9 branches. Distribution: central-1: 5, central-2: 3, central-3: 2 (mirrors real Bangkok logistics density).
- **D-03:** Hubs use real, well-known Thai logistics areas (geocodable by Google Maps Directions / Geocoding):
  - `hq-lat-krabang` — Express HQ — Lat Krabang Industrial Estate, Bangkok (central-1)
  - `branch-bang-na` — Express Branch — Bang Na, Bangkok (central-1)
  - `branch-nonthaburi` — Express Branch — Mueang Nonthaburi (central-1)
  - `branch-pathum-thani` — Express Branch — Mueang Pathum Thani (central-1)
  - `branch-samut-prakan` — Express Branch — Mueang Samut Prakan (central-1)
  - `branch-ayutthaya` — Express Branch — Phra Nakhon Si Ayutthaya (central-2)
  - `branch-nakhon-pathom` — Express Branch — Mueang Nakhon Pathom (central-2)
  - `branch-samut-sakhon` — Express Branch — Mueang Samut Sakhon (central-2)
  - `branch-ratchaburi` — Express Branch — Mueang Ratchaburi (central-3)
  - `branch-lop-buri` — Express Branch — Mueang Lop Buri (central-3)
- **D-04:** Single-leg routing only. Route = chosen hub → destination (one Google Maps Directions call). Internal HQ→branch transfer is operational cost, NOT customer-facing pricing. Matches Kerry/Flash/Thailand Post quoting model.

**Rate table re-architecture (D-05 through D-07)**
- **D-05:** Rate table extended to origin-zone × destination-zone matrix. New schema: `(shipping_type, origin_zone, dest_zone, weight_min_kg, weight_max_kg, base_rate_thb)`. Row count: 45 → 135 (3 origin × 3 dest × 3 ship × 5 weight). Lookup key = `(shipping_type, origin_zone, dest_zone, weight_kg)`. origin_zone derived from `hub_id` at lookup time via hub→zone map loaded from `hubs.json`.
- **D-06:** Symmetric distance-style multiplier matrix `M[origin_zone][dest_zone] = M[dest_zone][origin_zone]`. Diagonal cheapest; cross-zone scales with zone-distance. Generated by extended `generate_rate_table.py` with documented assumptions.
- **D-07:** `generate_rate_table.py` extended with new `ORIGIN_DEST_MULTIPLIER` 3×3 matrix constant. Existing `SHIPPING_TYPES`, `WEIGHT_TIERS`, `BASE_RATES` preserved. CSV gains `origin_zone` column placed BEFORE existing zone column (renamed `dest_zone`). `seed_database.py` updated for new schema.

**UI capture & default behavior (D-08 through D-10)**
- **D-08:** Hybrid origin capture — dropdown picker sets persistent default for the conversation; inline prose ("ship from Bang Na to Nonthaburi") overrides for that turn. Cold-start default = HQ Lat Krabang.
- **D-09:** When origin unspecified in both dropdown and prose, default to HQ silently. Pricing Agent's reasoning trace narrates: "Origin unspecified — defaulted to HQ Lat Krabang." User can override via dropdown or next-turn prose.
- **D-10:** Planner extracts `origin_hub_id` from user message — Gemini call already exists, hub shortlist injected into prompt, allowlist-validated, falls back to dropdown's value on invalid/missing. No extra RPM cost.

### Pattern 1: Additive Pydantic / TypedDict Extension (locked precedent)
**What:** New optional field added at end of existing schema; never reorder, never rename, never remove.
**When to use:** ALL state/model extensions in this phase.
**Examples in repo:**
- `AgentState.tool_call_count: Annotated[int, operator.add]` — Quick 260509-utd added this AFTER `guard_decision` (state.py:112)
- `AgentState.search_context: Optional[dict]` — Phase 5 D-11 added AFTER `approval_decision` (state.py:78)
- `RouteData` class — Pydantic model already permits `model_config` extension; adding `origin_hub_id: Optional[str] = None` at end is non-breaking.

**Example pattern:**
```python
# backend/agent/state.py — append AFTER tool_call_count
origin_hub_id: Optional[str]
"""Phase 999.9: chosen origin hub identifier (e.g., 'hq-lat-krabang',
'branch-bang-na'). Resolved by Planner from prose (D-10) or seeded from
ChatRequest.origin_hub_id (D-08). Pricing Agent reads this to derive
origin_zone via _load_hub_index() before calling lookup_rate.
None means dropdown not yet picked AND prose did not specify — Pricing
Agent silently defaults to 'hq-lat-krabang' (D-09) and narrates the
assumption in the reasoning trace."""
```

### Pattern 2: Allowlist-Validated LLM Extraction (locked precedent)
**What:** LLM emits a string field; Pydantic Literal or post-LLM validator confirms membership in the known set.
**When to use:** `origin_hub_id` in `PlannerOutput`.
**Examples in repo:**
- `PlannerOutput.next_step: Literal["fetch_fuel", "fetch_route", ...]` — invalid value triggers ValidationError → D-02 retry
- `PlannerOutput.user_intent: Literal["surcharge_query", ...]` — same Literal validation
- 999.1 fix in planner.py `_load_zone_index()` builds allowlist from JSON; same pattern for hubs

**Why Literal won't quite work for `origin_hub_id`:** 10 hub_ids in a Literal type works, but the source-of-truth is `hubs.json` (D-01). Hard-coding 10 strings into a Python `Literal[...]` duplicates the contract. **Recommended approach:** declare `origin_hub_id: Optional[str]` in PlannerOutput and validate against `_load_hub_index()` allowlist post-parse (mirrors how `_zone_for_destination` validates province against `_ZONE_INDEX`). On invalid: fall back to None (which then inherits from ChatRequest.origin_hub_id, then defaults to HQ).

**Example post-LLM validator:**
```python
# After parsed = _parse_structured(content), before the 999.1 merge:
if parsed.origin_hub_id is not None:
    valid_hub_ids = set(_load_hub_index().keys())  # cached via module-level _HUB_INDEX
    if parsed.origin_hub_id not in valid_hub_ids:
        logger.warning(
            "planner emitted invalid origin_hub_id=%r (allowed: %s); discarding",
            parsed.origin_hub_id, sorted(valid_hub_ids),
        )
        parsed.origin_hub_id = None
```

### Pattern 3: TTL Cache Key Extension (one-line change)
**What:** `TTLCache` keys are arbitrary tuples (see `_cache.py` — `_store: dict` keyed by anything hashable). Extending the key tuple is non-breaking.
**When to use:** `_route_cache` in `calculate_route.py`.
**Example:**
```python
# calculate_route.py:117 — current
key = (origin, destination)
# After 999.9
key = (origin_hub_id, destination)
```
The hub_id-resolved origin string is still passed to googlemaps.directions and stored in `RouteData.origin`, but the cache key uses `origin_hub_id` (the stable identifier) so two different prose strings that resolve to the same hub share a cache entry. Existing entries from Phase 1-8 will key-miss on the new schema (cold-start route fetch on first 999.9 turn), which is acceptable — TTL is 15 minutes anyway.

### Pattern 4: CSV-as-Source-of-Truth + SQLite-as-Cache (Phase 1 standard)
**What:** Generated CSV is the source of truth. SQLite is rebuilt from CSV via `seed_database.py`. Schema migration = re-run the seeder.
**When to use:** rate_table schema change.
**Why this works for 999.9:** `seed_database.py:38` already does `df.to_sql("rate_table", conn, if_exists="replace", index=False)`. When the new 135-row CSV is generated, re-running `seed_database.py` drops and recreates the table with the new column — no migration script needed. Per pandas 3.x docs (verified via WebSearch 2026-05-10): "`replace` completely drops and recreates the table with the new data structure."
**Idempotency:** Yes — `seed_database.py` is fully re-runnable. Multiple seeds produce the same result.

### Pattern 5: Hub Lookup Helper Mirroring Zone Index (locked precedent)
**What:** Module-level dict built once at import time from a JSON source.
**When to use:** Hub ID → zone resolution.
**Examples in repo:**
- `calculate_route.py:66` — `_ZONE_INDEX: dict[str, str] = _load_zone_index()` built once
- Same pattern for hubs:
```python
# backend/agent/tools/hubs.py (NEW)
_REPO_ROOT = Path(__file__).resolve().parents[3]
_HUBS_JSON = _REPO_ROOT / "data" / "raw" / "hubs.json"

def _load_hub_index() -> dict[str, dict]:
    """hub_id -> {name, address, zone}. Built once at import time.

    Returns:
        Dict keyed by hub_id with the full hub object as value. The address
        field is what's passed to googlemaps for geocoding/directions; the
        zone field is what's passed to lookup_rate as origin_zone.
    """
    with open(_HUBS_JSON, encoding="utf-8") as fh:
        return json.load(fh)

_HUB_INDEX: dict[str, dict] = _load_hub_index()


def origin_string_for(hub_id: str) -> str:
    """Resolve hub_id to the address googlemaps geocodes (D-04 contract)."""
    hub = _HUB_INDEX.get(hub_id)
    if not hub:
        raise ValueError(f"unknown hub_id={hub_id!r}; allowed={sorted(_HUB_INDEX)}")
    return hub["address"]


def origin_zone_for(hub_id: str) -> str:
    """Resolve hub_id to its zone (D-05 lookup_rate input)."""
    hub = _HUB_INDEX.get(hub_id)
    if not hub:
        raise ValueError(f"unknown hub_id={hub_id!r}; allowed={sorted(_HUB_INDEX)}")
    return hub["zone"]
```

### Pattern 6: Frontend Controlled Component + sessionStorage (Phase 4 + Phase 8 precedent)
**What:** React controlled component with sessionStorage for thread-scoped persistence; lifted state in ChatApp parent.
**When to use:** `HubPicker` component.
**Examples in repo:**
- `useChatStream.ts:115-123` reads `localStorage[LOCAL_STORAGE_KEYS.threadId]` post-mount (Pitfall 6 — avoid SSR hydration mismatch).
- ChatApp.tsx is the lifted-state parent for chat + conversations; HubPicker selection lives at this level.
**Decision rationale:** sessionStorage (NOT localStorage) for hub picker — selection should reset on browser tab close; localStorage would persist across sessions and surprise users. URL state would require Next.js search params, which conflicts with the SPA's single-route /chat structure.

**Example HubPicker:**
```typescript
// frontend/components/chat/HubPicker.tsx
'use client';
import { useEffect, useState } from 'react';
import clsx from 'clsx';

interface Hub {
  hub_id: string;
  name: string;
  address: string;
  zone: 'central-1' | 'central-2' | 'central-3';
}

interface Props {
  hubs: Hub[];
  value: string;
  onChange: (hubId: string) => void;
  disabled?: boolean;
}

const STORAGE_KEY = 'express_origin_hub_id';
const DEFAULT_HUB_ID = 'hq-lat-krabang';

export function HubPicker({ hubs, value, onChange, disabled }: Props) {
  return (
    <label className="flex items-center gap-2 text-sm text-text-primary">
      <span className="font-semibold">From:</span>
      <select
        value={value}
        onChange={(e) => onChange(e.target.value)}
        disabled={disabled}
        aria-label="Origin hub"
        className={clsx(
          'rounded border border-white/15 bg-white/5 backdrop-blur-md px-2 py-1',
          'text-text-primary focus:border-brand-via focus:outline-none focus:ring-2 focus:ring-brand-via/40',
          'disabled:opacity-60',
        )}
      >
        {hubs.map((h) => (
          <option key={h.hub_id} value={h.hub_id}>
            {h.name} ({h.zone})
          </option>
        ))}
      </select>
    </label>
  );
}

// In ChatApp.tsx, lifted state with sessionStorage seeding:
const [originHubId, setOriginHubId] = useState<string>(() => {
  if (typeof window === 'undefined') return DEFAULT_HUB_ID;
  return window.sessionStorage.getItem(STORAGE_KEY) ?? DEFAULT_HUB_ID;
});
useEffect(() => {
  if (typeof window === 'undefined') return;
  window.sessionStorage.setItem(STORAGE_KEY, originHubId);
}, [originHubId]);
```

The exact placement (header vs input row) and ARIA refinement is deferred to `/gsd:ui-phase 999.9`. The above is a working contract for the planner.

### Pattern 7: Cache-Aware Override Cascade (Phase 3 D-12 + Phase 5 D-08, EXTENDS cleanly)
**What:** When fuel_data and route_data are cached and fresh, planner skips fetch_fuel/fetch_route and goes straight to calculate_price.
**When to use 999.9:** The cache-skip cascade in `planner.py:355-369` keys route freshness on `_route_matches(state, merged_origin, merged_destination)` — origin/destination string match. **For 999.9 this needs to extend to also check `merged_origin_hub_id` matches the cached route_data's hub_id**, OR — simpler — keep the current string-match logic and rely on the cache key extension `(origin_hub_id, destination)` to handle hub-change cache invalidation at the TTLCache layer. **Recommendation:** simpler path — TTLCache layer handles invalidation, planner's `_route_matches` stays string-based. RouteData gains `origin_hub_id` for trace transparency, but `_route_matches` doesn't read it.

### Anti-Patterns to Avoid
- **Hard-coding hub list in Python** — hubs.json is the source of truth (D-01 mirror of zone_definitions.json). Do NOT duplicate the 10 hubs in `Literal[...]`, prompt strings, or constants.
- **Two-leg pricing** — explicitly out of scope (D-04 + Deferred Ideas). Single-leg only.
- **Renaming the existing `zone` column to `dest_zone` in Pydantic models or frontend types** — Pydantic `RouteData.zone` and frontend types already exist; renaming would cascade through 8 phases. **Only the rate_table CSV column renames** (because it's internal data, not a wire contract). RouteData.zone keeps its name (it's still the destination zone — same semantics).
- **Mutating live route cache entries with new key shape mid-deployment** — call `_route_cache.clear()` on backend startup, OR accept that a fresh `(origin_hub_id, destination)` tuple will cold-miss until populated. Cold-miss is fine (15-min TTL anyway).
- **Reading hubs.json on every request** — load once at import time (Pattern 5).

## Common Pitfalls

### Pitfall 1: Stale TTL Cache After Schema Change
**What goes wrong:** Backend starts with fresh code (new cache key shape `(origin_hub_id, destination)`) but the in-process `_route_cache` is empty anyway (resets on process restart per `_cache.py` docstring). HOWEVER, if the planner sends `origin_hub_id=None` for a turn (e.g., D-09 silent default not yet wired), `key = (None, destination)` is a valid tuple key — cache pollution with None keys.
**Why it happens:** Python `dict` accepts None as a key component. ChatRequest layer must seed the default BEFORE state hits planner.
**How to avoid:** ChatRequest handler in `backend/api/routes/chat.py` MUST default `req.origin_hub_id` to `"hq-lat-krabang"` when None at the API boundary, BEFORE state seeding. State only ever sees a valid hub_id.
**Warning signs:** Cache-hit logs with `(None, ...)` keys; route_data.origin = "Bangkok" (the historical D-05 fallback) on a fresh 999.9 turn.

### Pitfall 2: 999.1 Follow-Up Merge Doesn't Inherit `origin_hub_id`
**What goes wrong:** Phase 5 D-08 token-detection allow-list inherits `origin`, `destination`, `shipping_type`, `weight_kg` across follow-up turns. If `origin_hub_id` is not added to this list, follow-up turns lose the hub selection — the agent silently re-defaults to HQ on every "What about 25kg instead?" follow-up after the user picked Bang Na.
**Why it happens:** The planner `_PLANNER_BODY` and the followup_query inheritance logic (planner.py:251-294) are explicit per-field — `origin_hub_id` must be added to BOTH the prompt contract AND the post-LLM merge.
**How to avoid:**
1. Planner SYSTEM_PROMPT followup_query section gains `origin_hub_id` to the per-field null/inherit contract.
2. The 999.1 merge block (planner.py:296-307) gains `merged_origin_hub_id = parsed.origin_hub_id or state.get("origin_hub_id")`.
3. The followup_query null-out block (planner.py:251-294) gains an `origin_hub_id` token-detection clause: if the user message doesn't contain any of the 10 hub names AND prior state has a hub_id, null out parsed.origin_hub_id so prior wins.
**Warning signs:** Test "ship from Bang Na to Nonthaburi" → "What about 25kg?" returns total computed against `hq-lat-krabang` instead of `branch-bang-na`.

### Pitfall 3: Multiplier Matrix Diagonal Less Than 1.0 Inverts Pricing Story
**What goes wrong:** CONTEXT.md suggests diagonal ≈ 0.9x. With existing `ZONE_MULTIPLIERS = {central-1: 1.0, central-2: 1.25, central-3: 1.55}` already in `generate_rate_table.py:45-49`, the OLD pricing story was "central-1 is the cheapest base zone." If the new matrix has `M[central-1][central-1] = 0.9` AND the existing `ZONE_MULTIPLIERS[dest_zone]` is also still applied, you double-apply zone multipliers and the 0.9 diagonal lowers central-1 prices BELOW the existing v1.0 baseline — the rate table no longer matches docs.
**Why it happens:** Two competing zone-multiplier schemes (legacy single-zone destination multiplier + new 3×3 matrix) easy to combine accidentally.
**How to avoid:** **The new `ORIGIN_DEST_MULTIPLIER` matrix REPLACES the existing `ZONE_MULTIPLIERS` dict** — do not stack them. `generate_rate_table.py` becomes:
```python
# Phase 999.9 — REPLACES ZONE_MULTIPLIERS dict
ORIGIN_DEST_MULTIPLIER: dict[str, dict[str, float]] = {
    "central-1": {"central-1": 1.00, "central-2": 1.25, "central-3": 1.70},
    "central-2": {"central-1": 1.25, "central-2": 1.00, "central-3": 1.45},
    "central-3": {"central-1": 1.70, "central-2": 1.45, "central-3": 1.00},
}
```
**Numeric calibration recommendation (HIGH confidence — backed by existing v1.0 zone multipliers):**
- Diagonal = 1.00 (NOT 0.9 — keeps v1.0 central-1 rates byte-for-byte stable: 50, 55, 65, etc. for the lowest tier)
- one-zone-apart = 1.25 (matches existing central-2 multiplier exactly)
- two-zones-apart = 1.70 (close to existing central-3 = 1.55 but rounded up to reflect the COMBINED long-haul cost — old model only charged distance one-way; new model charges symmetric distance)

This calibration preserves the v1.0 demo's 50-698 THB rate range and central-1→central-1 diagonal exactly; the off-diagonals are slightly higher than v1.0 to reflect that long-haul shipments have a real origin-side cost the v1.0 model ignored. Document this assumption in the script header per D-07.

**Warning signs:** test_rate_table.py assertions on rate range (`50-698 THB` per Phase 1 STATE.md) start failing; central-1→central-1 5kg bounce no longer = 55 THB.

### Pitfall 4: Renaming `zone` Column In Place Breaks Existing Code Paths
**What goes wrong:** D-07 renames the CSV column from `zone` to `dest_zone`. If you also rename in Pydantic models, frontend types, RouteData, and lookup_rate's WHERE clause — you've cascaded through 8 phases of code. Tests assert on `route_data['zone']`. Pricing agent reads `state["route_data"]["zone"]` (pricing_agent.py:455). Frontend trace renders `route_data.zone`. Response node templates reference zone.
**Why it happens:** "zone" was always destination-zone in spirit; the rename only matters at the rate_table SQL boundary.
**How to avoid:** **Only rename the CSV header AND the SQL column.** Pydantic `RouteData.zone` keeps its name (still semantically the destination zone). The lookup_rate function signature uses parameter names `origin_zone` and `dest_zone`, but the SQL `WHERE zone = ?` clause becomes `WHERE dest_zone = ?` to match the renamed column. RouteData → lookup_rate adapter: `lookup_rate(shipping_type=ship, origin_zone=origin_zone_for(hub_id), dest_zone=route_data["zone"], weight_kg=weight)`.
**Warning signs:** Frontend trace panel renders `route_data.zone = undefined`; pricing_agent KeyError on "zone".

### Pitfall 5: Hub Shortlist Token Bloat in Planner Prompt
**What goes wrong:** Naively dumping all 10 hubs with full addresses + zones into SYSTEM_PROMPT adds ~400-600 tokens to every planner call. Gemini 2.0 Flash free tier is 15 RPM but token limits matter for response quality and cost (free tier has daily token limits in 2026).
**Why it happens:** Easy temptation to drop the entire hubs.json content into the prompt.
**How to avoid:** Inject a compact hub shortlist — `hub_id: name`, one per line, no addresses. Total ~150 tokens for 10 hubs. Example:
```python
# In planner.py prompt construction:
hub_shortlist = "\n".join(
    f"- {hub_id}: {hub['name']} ({hub['zone']})"
    for hub_id, hub in _HUB_INDEX.items()
)
HUB_INSTRUCTION = (
    "When the user mentions an origin hub by name, area, or branch identifier, "
    "extract `origin_hub_id` from this list:\n"
    + hub_shortlist
    + "\nReturn null if no hub is clearly mentioned. The dropdown's selection "
    "or default HQ Lat Krabang will be used as fallback."
)
```
Total prompt grows from ~1500 tokens (current) to ~1650 tokens — well within Gemini 2.0 Flash limits (32K input, 8K output).
**Warning signs:** Planner latency increases noticeably; quota errors on previously-passing turns.

### Pitfall 6: `req.origin_hub_id` Forwarded But Never Reaching State
**What goes wrong:** Frontend sends `origin_hub_id` in ChatRequest body. Backend Pydantic ChatRequest accepts it. But `_fresh_stream` in chat.py:218-225 builds `initial_state = {"messages": [{"role": "user", ...}]}` — only seeds `messages`. The new field never enters AgentState, so the planner reads `state.get("origin_hub_id")` → None → defaults to HQ on every turn regardless of dropdown.
**Why it happens:** `initial_state` is a literal dict with hard-coded keys.
**How to avoid:** `_fresh_stream` extends `initial_state`:
```python
initial_state = {
    "messages": [{"role": "user", "content": message}],
    "origin_hub_id": req.origin_hub_id or "hq-lat-krabang",  # D-08 cold-start default
}
```
**Warning signs:** Frontend dropdown picker has no observable effect on agent reasoning; trace always shows "Origin: hq-lat-krabang."

### Pitfall 7: HITL Threshold Fires Too Often Crashes Demo UX
**What goes wrong:** CONTEXT.md notes cross-zone shipments fire HITL more often (desirable). BUT: with `M[c-1][c-3] = 1.70`, a 50kg bounce shipment c-1 → c-3 might suddenly cross the 500 THB threshold where the v1.0 single-zone version did not. Demo flow that worked in v1.0 ("15kg Bounce Bangkok to Nonthaburi") now hits HITL approval on every long-haul example.
**Why it happens:** Threshold (500 THB) is unchanged; total magnitudes shift up because of new origin-zone amplification.
**How to avoid:** Validate against Phase 5 demo prompts. If existing demo turns now hit HITL more frequently than intended, EITHER (a) bump HITL_TOTAL_THB_THRESHOLD up (e.g., 700) — but this is a Phase 5 contract change and out-of-scope per CONTEXT.md, OR (b) keep threshold and update demo prompts in docs to reflect that HITL fires more often (which is the desirable demo behavior per CONTEXT.md "fire HITL more often, which is desirable").
**Recommendation:** (b). Document in the phase plan that demo prompts should still demonstrate sub-threshold (single-zone) AND supra-threshold (cross-zone) cases.
**Warning signs:** Test `test_hitl_gate.py` mock_pricing_low fixture (currently total=210 THB) needs revisit only if its rate inputs change; mock fixtures live independent of actual rate_table values, so this is pre-emptive only.

### Example 2: Extended generate_rate_table.py (D-07 conformant)
```python
"""Generate Express rate table CSV with documented assumptions.

Phase 999.9 (D-05/D-06/D-07): origin_zone × dest_zone matrix.

Assumptions (D-01 through D-07):
- 3 shipping types: bounce, retail_standard, retail_fast
- 3 zones: central-1 (Bangkok Metro), central-2, central-3
- 5 weight tiers: 0-5, 5-10, 10-20, 20-50, 50-999 kg
- Base rates in THB: ~50-500 range for central-1 dest
- ORIGIN_DEST_MULTIPLIER 3×3 symmetric matrix (D-06):
  M[origin_zone][dest_zone] = M[dest_zone][origin_zone]; diagonal = 1.0
  (preserves v1.0 central-1 dest rates byte-for-byte); off-diagonal
  scales with zone-distance.
- Final rate = base * M[origin_zone][dest_zone], rounded to nearest int.
- 135 rows = 3 origin × 3 dest × 3 ship × 5 weight (D-05).
"""
SHIPPING_TYPES: list[str] = ["bounce", "retail_standard", "retail_fast"]
ZONES: list[str] = ["central-1", "central-2", "central-3"]
WEIGHT_TIERS: list[tuple[int, int]] = [(0, 5), (5, 10), (10, 20), (20, 50), (50, 999)]
BASE_RATES: dict[str, list[int]] = {
    "bounce": [55, 85, 130, 220, 380],
    "retail_standard": [50, 75, 110, 190, 340],
    "retail_fast": [65, 100, 155, 260, 450],
}
# Phase 999.9 D-06: REPLACES legacy ZONE_MULTIPLIERS dict.
ORIGIN_DEST_MULTIPLIER: dict[str, dict[str, float]] = {
    "central-1": {"central-1": 1.00, "central-2": 1.25, "central-3": 1.70},
    "central-2": {"central-1": 1.25, "central-2": 1.00, "central-3": 1.45},
    "central-3": {"central-1": 1.70, "central-2": 1.45, "central-3": 1.00},
}

def main() -> None:
    rows: list[dict] = []
    for shipping_type in SHIPPING_TYPES:
        for origin_zone in ZONES:
            for dest_zone in ZONES:
                multiplier = ORIGIN_DEST_MULTIPLIER[origin_zone][dest_zone]
                for i, (weight_min, weight_max) in enumerate(WEIGHT_TIERS):
                    base = BASE_RATES[shipping_type][i]
                    rate = round(base * multiplier)
                    rows.append({
                        "shipping_type": shipping_type,
                        "origin_zone": origin_zone,
                        "dest_zone": dest_zone,
                        "weight_min_kg": weight_min,
                        "weight_max_kg": weight_max,
                        "base_rate_thb": rate,
                    })
    df = pd.DataFrame(rows)
    output_path = Path(__file__).parent.parent / "raw" / "express_rate_table.csv"
    df.to_csv(output_path, index=False)
    print(f"Generated {len(df)} rate table rows -> {output_path}")  # 135
```

### Example 3: Extended lookup_rate (D-05 conformant)
```python
def lookup_rate(
    shipping_type: str,
    origin_zone: str,
    dest_zone: str,
    weight_kg: float,
) -> RateResult:
    """Look up base rate from rate_table (Phase 999.9 D-05 schema).

    Args:
        shipping_type: "bounce" | "retail_standard" | "retail_fast".
        origin_zone: "central-1" | "central-2" | "central-3" — derived
            from origin hub_id via origin_zone_for(hub_id).
        dest_zone: "central-1" | "central-2" | "central-3" — destination
            zone (RouteData.zone).
        weight_kg: Positive float; valid range (0, 999).

    D-13 half-open intervals + C-02 sentinel weight_max_kg=999 still apply.

    Raises:
        ValueError: On unknown ship/origin_zone/dest_zone/weight.
    """
    if weight_kg <= 0:
        raise ValueError(f"weight_kg must be positive, got {weight_kg}")

    query = """
        SELECT base_rate_thb, weight_min_kg, weight_max_kg
        FROM rate_table
        WHERE shipping_type = ?
          AND origin_zone = ?
          AND dest_zone = ?
          AND weight_min_kg <= ?
          AND ? < weight_max_kg
        LIMIT 1
    """
    with sqlite3.connect(_DB_PATH) as conn:
        row = conn.execute(
            query,
            (shipping_type, origin_zone, dest_zone, weight_kg, weight_kg),
        ).fetchone()
    if row is None:
        raise ValueError(
            f"No rate found for shipping_type={shipping_type!r}, "
            f"origin_zone={origin_zone!r}, dest_zone={dest_zone!r}, "
            f"weight_kg={weight_kg}"
        )
    base_rate, wmin, wmax = row
    tier = f"{wmin}-{wmax}kg" if wmax < 999 else f"{wmin}+kg"
    return RateResult(base_rate=float(base_rate), currency="THB", rate_tier=tier)
```

## Open Questions

1. **Should `RouteData.origin` continue to be the geocoded address string OR transition to `origin_hub_id`?**
   - What we know: keeping both as separate fields (origin = address; origin_hub_id = hub key) preserves trace richness — frontend can render "Lat Krabang Industrial Estate" while still knowing the canonical hub.
   - What's unclear: Pydantic `RouteData.origin` was a `str` (not Optional) in v1.0; if we keep it, it stays str (still populated). No backwards-compat issue.
   - Recommendation: Keep `origin: str` (now populated from hub.address) AND ADD `origin_hub_id: Optional[str]`. Both flow through trace UI.

2. **Should the silent-default-to-HQ narration (D-09) be added by the planner or the pricing agent?**
   - What we know: D-09 says "Pricing Agent's reasoning trace narrates the assumption."
   - What's unclear: technically the planner could also append a trace bullet when it detects `state.origin_hub_id is None` after extraction.
   - Recommendation: Pricing Agent — matches D-09 verbatim and `_build_bullets` is already structured for additive bullets. Add a new bullet: "Origin unspecified — defaulted to HQ Lat Krabang." prepended when `state.get("origin_hub_id")` is None at pricing-agent entry.

3. **Should `_load_hub_index()` cache be invalidated if hubs.json changes at runtime?**
   - What we know: zone_definitions.json's `_ZONE_INDEX` is loaded once at import time and never invalidated.
   - What's unclear: dev workflow might want hot-reload; production demo doesn't.
   - Recommendation: Match `_ZONE_INDEX` precedent — load once at import. Restart uvicorn on hubs.json change (already a known pattern per Quick 260509-rs8 STATE.md note).

4. **Where exactly should the HubPicker render — header bar, input row, or sidebar?**
   - What we know: CONTEXT.md explicitly defers placement to `/gsd:ui-phase 999.9`.
   - What's unclear: nothing — this is correctly deferred.
   - Recommendation: Phase plan should include a `/gsd:ui-phase 999.9` task as its UI subphase. Backend phase work proceeds independent of placement decision.

5. **Should existing v1.0 demo prompts in `docs/` be updated to use hub-prefixed prose?**
   - What we know: docs/screenshots/ exist for v1.0 happy path; demo.mp4 not yet recorded (per STATE.md).
   - What's unclear: whether 999.9 supersedes the v1.0 demo content.
   - Recommendation: Phase plan should include doc updates (architecture.md + README.md hub network section) as a Wave N task, but NOT update screenshots — those need v1.1 milestone re-recording.
