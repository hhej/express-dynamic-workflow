# Phase 999.9 — UI Design Contract

> Visual and interaction contract for Phase 999.9 (HQ/Branch Origin Model). Generated by gsd-ui-researcher, verified by gsd-ui-checker. All decisions trace to CONTEXT.md (D-01..D-10), RESEARCH.md (Patterns 1–6, esp. Pattern 6 controlled-component+sessionStorage), the locked Phase 4 UI-SPEC tokens, and the dark cosmic glass-morphism theme (quick-260509-e0p) currently shipping in `frontend/app/globals.css`.

The phase ships exactly one new visual surface — `HubPicker` (origin hub dropdown) — plus a one-bullet narration line in the existing reasoning trace. Everything else (chat layout, trace panel, dashboard, sidebar) is unchanged. This contract therefore inherits Phase 4's tokens verbatim and adds only the new component's design rules and the new copy strings.

## Visual Hierarchy

**Primary focal point (unchanged from Phase 4):** the live reasoning trace panel — the agent's visible reasoning is the product. In Phase 999.9 the trace becomes *more* informative: when origin is unspecified, the Pricing Agent emits a narration bullet ("Origin unspecified — defaulted to HQ Lat Krabang") that makes the assumption visible (D-09). This is a content-level upgrade to an existing focal point, not a new focal point.

**Secondary anchor (unchanged):** the chat input row — `ChatInput` textarea + `Send` button. The accent send button stays the brightest at-rest element on screen.

**New tertiary anchor:** the `HubPicker` dropdown — sits *adjacent to but visually subordinate to* the chat input row. It must read as a quiet, persistent default-selector — not a primary action. Glass-surface treatment with no accent fill at rest. The "From:" label anchors it semantically; the active `<option>` text reads at body weight, not bold.

**Recede:** sidebar (left, `w-64`), trace panel (right, `w-96`) — unchanged.

Inherited from Phase 4 UI-SPEC §Spacing Scale verbatim. All multiples of 4, mapped to Tailwind v4 default scale.

| md | 16px | `gap-4`, `p-4` | Padding inside the input-row container that holds HubPicker + ChatInput (matches existing `ChatInput` form `p-4`) |

| 2xl | 48px | `gap-12` | (reserved, unused) |
| 3xl | 64px | `gap-16` | (reserved, unused) |

**Layout (unchanged from Phase 4):** three-column desktop (sidebar 256px / chat flex-1 / trace panel 384px). Mobile breakpoint <768px collapses rails to drawers.

**HubPicker placement (locked decision — see Pattern 6 in RESEARCH.md):** HubPicker renders **inside the chat input row container**, **above** the existing `ChatInput` textarea, on its own line. Selected because:
1. The picker is *part of the user's outgoing message contract* (origin = where this shipment ships from). Co-locating it with the input row makes that contract visible at the moment the user composes their question.
2. Header bar placement was rejected because the chat header doesn't exist as a discrete surface today (the three-column layout has no top bar); inventing one for a single 10-option control would add layout debt.
3. Above-the-textarea (vs. inline-left of the textarea) was selected so the textarea retains its full row width for prose, and the HubPicker has clean horizontal room to render the longest hub label ("Express Branch — Phra Nakhon Si Ayutthaya (central-2)" ≈ 56 chars) without truncation on desktop.

Concrete spacing: input-row container becomes a `flex flex-col gap-2 border-t border-white/10 p-4` (matches existing `ChatInput`'s top border + padding). HubPicker is the first child (`<label class="flex items-center gap-2">` with the `<select>`), the existing `<form>` is the second child.

**Touch target:** the `<select>` element renders at native height ~32px on desktop browsers and ~44px on iOS/Android — meets WCAG 2.5.5 (Target Size, Level AAA 44×44 CSS px) on touch devices and the AA 24×24 minimum on desktop. No exception needed.

**Exceptions:** none.

Inherited from Phase 4 UI-SPEC §Typography verbatim. 3 sizes, 2 weights — meets the 3–4 / 2 cap.

|------|------|--------|-------------|----------------|----------------------|

**Notes:**

- No new size or weight is introduced. The 3-size / 2-weight cap holds.

### Inherited copy (Phase 4 UI-SPEC §Copywriting — unchanged)

App title, sidebar heading, "+ New conversation" button, primary CTA aria-label, trace panel heading, empty states, status badges, capped callout copy, clarify and partial card headings, dashboard headings, feedback button labels, SSE error toast, network failure copy — **all unchanged**.

### Voice and tone rules (inherited, locked)

- Second-person ("you") in instructional copy. Never first-person.
- Sentence case for headings and buttons (e.g., "From:", not "FROM:" or "From").
- Error copy follows "What happened. What you can do." (e.g., "Couldn't load hubs. Defaulting to HQ Lat Krabang.")
- "Bangkok Metro" is the user-facing region phrase. Hub display strings include real Thai place names ("Lat Krabang", "Bang Na", "Mueang Nonthaburi", "Phra Nakhon Si Ayutthaya") in their canonical romanizations — these are fixed by D-03.
- "HQ" and "Branch" are the user-facing role distinguishers. Internal `hub_id` strings (`hq-lat-krabang`, `branch-bang-na`) appear only in trace JSON detail panes; they are technical, not user-facing.

### Destructive actions

**None in Phase 999.9.** HubPicker is a non-destructive selector — picking a different hub does not delete any data, does not invalidate any past conversation, and has no irreversible consequence. A change merely updates the persistent default for the conversation; the next-turn prose can still override. No confirmation dialog is required.

## Interaction Contracts

The "this is how it works" rules. Executor must preserve verbatim.

| Interaction | Contract |
|------------|----------|
| Cold start (D-08) | On first mount: `originHubId` initializes to `'hq-lat-krabang'`. sessionStorage key `express_origin_hub_id` is read post-hydration (avoid SSR mismatch — RESEARCH.md Pitfall 6) and overrides the default if present. |
| Hub change (D-08 persistent default) | User picks a hub from the `<select>` → `onHubChange(hub_id)` fires → ChatApp updates `originHubId` state AND writes sessionStorage → next `send()` includes `origin_hub_id` in the POST body. Picker selection persists across turns within the same browser tab; resets on tab close (sessionStorage scope). |
| Prose override (D-08 per-turn override) | User types "ship from Bang Na to Nonthaburi" → request body still carries the dropdown-selected `origin_hub_id` (the FE does not parse prose) → Planner extracts `origin_hub_id='branch-bang-na'` from the message via the hub-shortlist injection (RESEARCH.md Pattern 2) → server-side `parsed.origin_hub_id` (when valid) wins over `state.origin_hub_id` for that turn → next turn returns to the dropdown's value unless the user picks again or another prose override fires. The dropdown selection is **not visually changed** by a prose override (the dropdown is the *persistent* default; the override is *per-turn*). |
| Silent default-to-HQ (D-09) | If `origin_hub_id` is absent from both dropdown (uninitialized — should never happen post-cold-start) AND prose, Pricing Agent uses `'hq-lat-krabang'` and emits the bullet "Origin unspecified — defaulted to HQ Lat Krabang." in its reasoning trace. The bullet is rendered in the Pricing Agent trace step's existing Output pane via the existing PricingReasoning.bullets pipeline (quick-260509-uwb) — no new trace UI surface. |
| Disabled state | `disabled` is true when `chat.status === 'streaming' \|\| chat.status === 'awaiting_approval'` — same condition as ChatInput's `inputDisabled` prop. The HubPicker `<select>` reads the same prop chain as ChatInput. |
| Hub fetch failure | `lib/hubs.ts` (or static-import equivalent — see Claude's Discretion below) fetches the hub list on app mount. On failure: render the picker with one disabled `<option>` reading "Hubs unavailable", plus an inline `role="alert"` line "Couldn't load hubs. Defaulting to HQ Lat Krabang." Server still operates — the request omits `origin_hub_id` and the backend defaults to HQ at the API boundary (RESEARCH.md Pitfall 1). |
| Resume conversation interaction | When the user clicks a thread in the sidebar to resume: `chat.threadId` is set, `messages` are replayed. **The dropdown does NOT inherit the resumed conversation's last-known hub** — it stays at the user's current sessionStorage value. Rationale: the dropdown represents "what hub the user is currently shipping from", not "what was used in the past". A user resuming an old conversation about a Bang Na shipment from a new tab where they picked Lat Krabang should see Lat Krabang in the dropdown, not be silently switched. The next turn's request will use the current dropdown value, which the planner can still override via prose. |
| Keyboard semantics | Native `<select>` browser semantics: `Tab` focuses, `Space` or `Enter` opens the option list, arrow keys navigate, `Enter` selects. No custom keyboard handler — relying on native is the WCAG-safest path and adds zero JS. |
| ARIA semantics | `<label htmlFor="hub-picker">From:</label>` paired with `<select id="hub-picker" aria-label="Origin hub">` — both label association AND aria-label are present, defense-in-depth for screen readers. `aria-disabled` follows the `disabled` attribute. No `role` override (native `<select>` already announces as combobox/listbox). On error: the inline error line uses `role="alert"` to announce on render. |
| Mobile breakpoint (D-05 inherited) | <768px: HubPicker stays visible *above* the textarea (input row stacks vertically anyway via `flex-col`), so no responsive change is required. The `<select>` renders as the OS native picker on mobile (iOS wheel, Android dropdown), which is the expected behavior. |

### Claude's Discretion items resolved by this UI-SPEC

CONTEXT.md §Claude's Discretion explicitly defers three items to `/gsd:ui-phase 999.9` — this UI-SPEC resolves all three:

1. **Dropdown placement (header bar vs input row):** RESOLVED → **inside the chat input row container, above the existing ChatInput textarea, on its own line.** Rationale in §Spacing Scale.
2. **Styling:** RESOLVED → **glass-surface treatment matching the ChatInput textarea exactly.** No accent fill. Focus ring `brand-via/40`. Disabled `opacity-60`. Rationale in §Color → "HubPicker color rules".
3. **Exact ARIA / keyboard semantics:** RESOLVED → **native `<select>` semantics + paired `<label htmlFor>` + belt-and-braces `aria-label="Origin hub"` + `role="alert"` on the error line.** No custom keyboard handler. Rationale in §Interaction Contracts → "Keyboard semantics" + "ARIA semantics".

### Open Discretion item passed to planner

- **Hub fetch transport:** static-import (`import hubs from '@/data/hubs.json'` at build time, served from `frontend/data/hubs.json` mirrored from `data/raw/hubs.json`) vs. runtime fetch from a new `GET /api/hubs` endpoint. Both support the contract above. Static-import is simpler (one less endpoint, one less mount-time fetch, no failure mode for "hubs.json missing"); runtime fetch is more flexible (hub list can change without a frontend redeploy). RESEARCH.md does not lock this choice. Planner picks during `/gsd:plan-phase` based on whether hub list churn is expected. **The HubPicker component itself doesn't care** — it accepts `hubs: Hub[]` as a prop either way.
