### sessionStorage seeding flow (post-hydration — Pitfall 6)

```
Cold render:
  useState(DEFAULT_HUB_ID)  →  originHubId = 'hq-lat-krabang'

Post-mount useEffect (deps=[]):
  read window.sessionStorage['express_origin_hub_id']
  IF stored AND HUB_ID_ALLOWLIST.has(stored)
    setOriginHubId(stored)  →  re-render with stored value
  ELSE
    no-op (allowlist guard — invalid stored values silently fall back to default)

User picks a hub:
  HubPicker.onChange  →  handleHubChange(hubId)  →  setOriginHubId(hubId)

Persistence useEffect (deps=[originHubId]):
  window.sessionStorage.setItem('express_origin_hub_id', originHubId)
```

**Why post-hydration:** SSR renders DEFAULT_HUB_ID (sessionStorage is browser-only). If we read sessionStorage during initial useState, the server-rendered HTML and client-hydrated React would mismatch on the first render — Pitfall 6. Reading post-mount means the cold render is identical SSR/CSR, then a single re-render happens to apply the stored value. Visually negligible flicker (one tick).

### UI-SPEC §"Open Discretion item passed to planner" resolution

**Static-import chosen** over runtime fetch from a `GET /api/hubs` endpoint.

**Rationale:**
- Simpler — one less endpoint, one less mount-time fetch, no failure mode for "hubs.json missing"
- Hub list churn is unlikely in v1.1 (UI-SPEC §Hub display strings locks the 10 hubs verbatim)
- No new MSW handler needed in tests; static-import resolution works the same in test/dev/prod
- TypeScript `resolveJsonModule: true` already configured in `frontend/tsconfig.json`

**Trade-off:** any future hub-list change requires a frontend redeploy (vs. backend-only redeploy with runtime fetch). Acceptable for demo-grade. The duplication between `data/raw/hubs.json` and `frontend/data/hubs.json` is documented here so a future phase can centralize (e.g., a prebuild script that copies one to the other, or extraction to a shared `@express/hubs` package).
