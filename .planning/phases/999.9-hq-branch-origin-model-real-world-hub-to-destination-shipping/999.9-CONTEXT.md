# Phase 999.9: HQ/Branch Origin Model - Context

**Gathered:** 2026-05-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the implicit-origin model (free-text origin string + destination-only zone pricing) with an explicit hub-based origin: sender chooses HQ or one of N branches via dropdown OR prose, and the agent calculates route + surcharge from the actual chosen hub to the destination. Matches how Kerry/Flash/Thailand Post quote shipments to customers. Bangkok Metro scope only — regional expansion is Phase 999.8.

</domain>

<decisions>
## Implementation Decisions

### Hub data model & seed

- **D-01:** Hub represented as object: `{hub_id, name, address, zone}`. Seeded from `data/raw/hubs.json` (mirrors the `data/raw/zone_definitions.json` shape pattern — top-level keyed object with named child fields).
- **D-02:** 10 hubs total: 1 HQ + 9 branches, distributed by zone density to mirror real Bangkok logistics density (central-1: 5, central-2: 3, central-3: 2).
- **D-03:** Hubs use real, well-known Thai logistics areas (geocodable by Google Maps Directions / Geocoding):
  - `hq-lat-krabang` — Express HQ — Lat Krabang Industrial Estate, Bangkok (central-1)
  - `branch-bang-na` — Express Branch — Bang Na, Bangkok (central-1)
  - `branch-nonthaburi` — Express Branch — Mueang Nonthaburi (central-1)
  - `branch-pathum-thani` — Express Branch — Mueang Pathum Thani (central-1)
  - `branch-samut-prakan` — Express Branch — Mueang Samut Prakan (central-1)
  - `branch-ayutthaya` — Express Branch — Phra Nakhon Si Ayutthaya (central-2)
  - `branch-nakhon-pathom` — Express Branch — Mueang Nakhon Pathom (central-2)
  - `branch-samut-sakhon` — Express Branch — Mueang Samut Sakhon (central-2)
  - `branch-ratchaburi` — Express Branch — Mueang Ratchaburi (central-3)
  - `branch-lop-buri` — Express Branch — Mueang Lop Buri (central-3)
- **D-04:** Single-leg routing: route = chosen hub → destination (one Google Maps Directions call). The internal HQ→branch transfer is operational cost, NOT customer-facing pricing. Matches how Kerry/Flash/Thailand Post quote shipments to end customers.

### Rate table re-architecture

- **D-05:** Rate table extended to origin-zone × destination-zone matrix. New schema: `(shipping_type, origin_zone, dest_zone, weight_min_kg, weight_max_kg, base_rate_thb)`. Row count grows from 45 → 135 rows (3 origin-zones × 3 dest-zones × 3 ship types × 5 weight tiers). Lookup key = `(shipping_type, origin_zone, dest_zone, weight_kg)`. Origin-zone is derived from `hub_id` at lookup time via a hub→zone map loaded from `hubs.json`.
- **D-06:** Pricing philosophy: symmetric distance-style multiplier matrix `M[origin_zone][dest_zone] = M[dest_zone][origin_zone]`. Diagonal (intra-zone) is cheapest; cross-zone scales with zone-distance. Generated by extended `data/scripts/generate_rate_table.py` with documented assumptions in the script header (D-01..D-04 style).
- **D-07:** `data/scripts/generate_rate_table.py` extended to emit the 135-row matrix using a new `ORIGIN_DEST_MULTIPLIER` 3×3 matrix constant. Existing constants (`SHIPPING_TYPES`, `WEIGHT_TIERS`, `BASE_RATES`) preserved. CSV schema gains an `origin_zone` column placed before the existing zone column (renamed `dest_zone`). `data/scripts/seed_database.py` updated for the new schema.

### UI capture & default behavior

- **D-08:** Hybrid origin capture: a dropdown picker in the chat UI sets a persistent default origin for the conversation; inline prose ("ship from Bang Na to Nonthaburi") overrides for that turn. Default dropdown selection on cold start = HQ Lat Krabang.
- **D-09:** When origin is unspecified in both dropdown and prose, agent defaults to HQ silently. Pricing Agent's reasoning trace narrates the assumption in a bullet: "Origin unspecified — defaulted to HQ Lat Krabang." User sees the assumption and can override via dropdown or next-turn prose.
- **D-10:** Prose extraction approach: Planner (Gemini 2.0 Flash) extracts `origin_hub_id` from the user message. Planner prompt is updated to inject the 10 hub_ids + names + areas as a shortlist; Gemini returns `origin_hub_id` directly in the existing extraction JSON. Planner validates the returned id against the allowed-list; on invalid/missing, falls back to the dropdown's current value. Reuses the existing planner extraction Gemini call — no extra RPM cost.

### Claude's Discretion

- Exact multiplier values in the symmetric `M[origin_zone][dest_zone]` matrix. Planner can derive from documented assumptions (e.g., diagonal ≈ 0.9x, one-zone-apart ≈ 1.25x, two-zones-apart ≈ 1.7x as starting point).
- Cache key shape for the route TTL cache: extending key from `(origin, destination)` to `(origin_hub_id, destination)` is straightforward.
- AgentState field shape: add `origin_hub_id: Optional[str]` alongside (not replacing) existing `origin: Optional[str]` — backwards-compat with Phase 1-8 contracts. Resolve `origin_hub_id` → origin string via `hubs.json` lookup before passing to `calculate_route`.
- Pricing Agent / `lookup_rate` tool signature: extend to `lookup_rate(shipping_type, origin_zone, dest_zone, weight_kg)` — additive change.
- Dropdown placement (header bar vs input row), styling, and exact ARIA/keyboard semantics — defer to `/gsd:ui-phase 999.9` for the UI design contract.
- Follow-up turn inheritance: extend Phase 5 D-08 token-detection allow-list to include `origin_hub_id` (inherited from prior turn unless explicitly changed in prose or via dropdown).
- HITL approval gate threshold: keep current Phase 5 fixed THB threshold; cross-zone shipments will fire HITL more often, which is desirable.

<specifics>
## Specific Ideas

- "Like how Kerry / Flash / Thailand Post quote shipments — sender picks a drop-off branch, system quotes from there." Single-leg pricing, not two-leg.
- 10 hubs felt right because "5 was kinda not impressive enough" for a real Express network — central-1 dense (5), central-2 medium (3), central-3 light (2) mirrors real Bangkok logistics density.
- Lat Krabang HQ specifically because it's a real, well-known Thai e-commerce / logistics cluster (Kerry, Flash, Lazada, Shopee all have facilities there) — strongest "this looks like a real network" signal for graders.
- Default to HQ silently with a trace-narration line — user sees the assumption and can override; matches Kerry/Flash where unspecified-origin defaults to a designated branch.

</specifics>

<deferred>
## Deferred Ideas

- **Regional expansion** (Northeast / North / South EPPO baselines, expanded zones) → Phase 999.8 (already in backlog)
- **Two-leg pricing model** (HQ → branch → destination, summed legs) → out of scope; would change the customer-facing pricing narrative
- **Per-hub rate table** (origin = `hub_id` not `origin_zone`) → out of scope; would explode the rate table to N×3×3×5 = 450 rows for 10 hubs and obscure the "zones drive pricing" demo story
- **Hub network expansion to 15+** (more central-1 density, Suvarnabhumi airport hub) → captured for future iteration once v1.1 lands and we know whether 10 is enough for the demo
- **Origin-distance kicker on top of dest-zone base** (rejected option B in rate-table area) → kept here in case post-launch empirical pricing review prefers the simpler approach
- **Asymmetric labor-economics multiplier matrix** (rejected option B in pricing-philosophy area) → kept here in case future scope wants origin-density-aware pricing
- **Cache invalidation & state migration deep-dive** — user opted to skip this gray area; planner / researcher will resolve from D-05 / D-08 / D-09 / D-10 + Phase 3 D-12 + Phase 5 D-08 conventions
