# Phase 5 — UI Design Contract

> Visual and interaction contract for Phase 5 (Polish, Observability & Docs). Generated by gsd-ui-researcher, verified by gsd-ui-checker. **Phase 5 INHERITS the Phase 4 contract verbatim** (`.planning/phases/04-frontend-reasoning-trace/04-UI-SPEC.md`) and EXTENDS only where Phase 5 adds net-new user-facing surface — the HITL approval card (D-06/D-07), the search-context summary line above markdown (D-11), two new trace-step agent labels (`hitl_gate`, `search_agent` per D-08/D-10), and the feedback wire swap (D-16, no visual change). All Phase 4 tokens, palette, copywriting voice, and interaction rules carry through unchanged.

## Visual Hierarchy (Phase 5 deltas)

**Phase 4 baseline preserved:** the live reasoning trace panel (right rail, 384px) remains the primary focal point; chat input row (bottom of center column) is the secondary anchor; sidebar (left, 256px) is the tertiary anchor that recedes at rest.

**Phase 5 additions:**

- **HITL approval card** is a *blocking inline element* in the assistant message slot — the most attention-demanding surface in the chat column when present. Yellow-50/yellow-300 visual treatment matches the locked Phase 4 `capped` callout palette (the only Phase 4 color that signals "needs human review"); reusing the palette tells the user "this is a review-required moment" without adding a new color category. Approve and Deny buttons share top-of-card visual weight; Approve is **NOT** styled with the accent blue — that would imply "default action," but D-07 requires the user to make an informed decision. Both buttons use the secondary outline treatment with explicit text labels.
- **Search-context summary line** is a *secondary element* prepended above the markdown prose answer when `final_payload.search_context.summary` is present. It uses 12px caption type on `bg-gray-50` so it reads as "context provenance," not as the primary answer. The user's eye flows: search context → prose → breakdown table → feedback buttons.
- **Trace panel parallel-fan-out demo** — when `fuel_agent` and `route_agent` trace entries arrive in the same superstep (D-01), they render as adjacent list items with timestamps within ~50ms. No layout change; the visible-overlap timestamps in the expanded detail are the demo evidence (ROADMAP §Phase 5 success criterion 1). No special "parallel" badge — the audit trail is the timestamp itself.

### Voice and tone rules (Phase 4 inherited; Phase 5 additions)

- **HITL approval prose:** neutral and informative, never alarming. "Approval required" not "Warning" or "Alert"; "This surcharge total exceeds the review threshold" not "Suspicious shipment detected".
- **Search-context provenance:** the "Market context:" prefix is non-negotiable — it tells the user the line is search-derived, distinguishing it from the deterministic formula. This is a transparency commitment that supports the CLAUDE.md Core Value: "visible reasoning is what makes this agentic, not just automated."
- **Bangkok Metro applies to all Phase 5 NEW user-facing copy:** ApprovalCard prose, search-context summary captions, README, architecture.md, data-sources.md, demo.mp4 captions. Internal `central-1/2/3` zone IDs in trace JSON detail panes are NOT user-facing (they are debug/audit-only).

### Destructive confirmations

**None in Phase 5.** Approve and Deny in the ApprovalCard are *deliberative*, not *destructive* — they finalize a recommendation but do not delete or irreversibly mutate anything (the conversation history persists in the checkpointer either way; a denied surcharge can be re-asked with adjusted inputs). No confirm-dialog interstitial is required. If a future phase adds destructive actions (delete conversation, etc.), the standard pattern from Phase 4 applies (confirm dialog with the action verb echoed).

## Interaction Contracts (Phase 5 additions)

Behaviours the executor must preserve verbatim. Phase 4 contracts (chat send, trace expand/collapse, new conversation, resume conversation, capped result, range toggle, SSE error, mobile breakpoint) carry through unchanged.

| Interaction | Contract |
|-------------|----------|
| **HITL approval gate fires (D-04, D-05, D-06)** | When SSE event `approval_required` arrives mid-stream: `useChatStream` flips status to `awaiting_approval` and stores the payload (`{thread_id, surcharge_result, threshold}`); `MessageList` swaps the in-flight assistant slot for `ApprovalCard`; chat input is **disabled** (treat `awaiting_approval` like `streaming` for input gating); trace panel keeps showing the current-turn entries including the `hitl_gate` step with `status="warn"` badge. |
| **HITL approve clicked (D-06, D-07)** | Both Approve/Deny buttons disable; `ApprovalCard` shows "Sending your decision…" caption; `useChatStream.approve(threadId, true)` POSTs `/api/chat` with `{thread_id, approve: true}`; the resume response streams a fresh trace + final answer; on `done`, `MessageList` replaces the `ApprovalCard` slot with the final `MarkdownAnswer` (the standard `ok`-status render). The approval-required entry remains visible in the trace (Phase 4 D-08 current-turn rule applies — past-turn replay is still deferred). |
| **HITL deny clicked (D-07)** | Same as approve, but `approve: false`. The resume response renders a `PartialCard` with the deny-path prose (no breakdown table). The original `surcharge_result` is NOT discarded from state — it remains in the checkpointer so a future Langfuse audit can reconstruct what was offered and rejected — but it is NOT shown in the chat (deferred decision: "Show declined recommendation" affordance is not implemented in v1 per CONTEXT.md §Claude's Discretion). |
| **HITL resume failure** | If the resume `POST /api/chat` fails (network or 5xx), re-enable Approve/Deny so the user can retry. `console.error` the failure; emit an inline "Couldn't send your decision. Try again." error message inside the `ApprovalCard` (red-600 text, no toast — toast is reserved for SSE-error per D-19). |
| **Search-context line render (D-11)** | When `final_payload.search_context.summary` is non-empty: render `SearchContextLine` ABOVE the `prose` block in `MarkdownAnswer`. When `search_context` is `null` (Tavily failure per D-12): omit the line entirely; render only the markdown prose. The cap-line strip-the-line logic from Phase 4 D-11 runs UNCHANGED — `SearchContextLine` renders between `CapCallout` (if any) and the prose, never inside the markdown stream. |
| **Search-context source expand** | Clicking the "Sources: {n}" caption toggles a `<details>`/`<summary>` block listing each source as `{title} — {published_at}` (title as link, opens in new tab). Keyboard `Enter`/`Space` activate. Default state is collapsed. When `sources.length === 0`, the line is not rendered. |
| **Trace step for `hitl_gate` (D-08)** | Pre-pause entry: `agent="hitl_gate"`, `tool="interrupt"`, `tool_input={threshold, total}`, `tool_output={decision_pending: true}`, `status="warn"`. Post-resume entry (separate stream): `agent="hitl_gate"`, `tool="interrupt"`, `tool_output={decision: "approve"\|"deny"}`, `status="ok"`. Both render through the existing `TraceStep` component using the new `AGENT_LABEL["hitl_gate"] = "Approval gate"` mapping; no special-case render. |
| **Trace step for `search_agent` (D-10)** | One D-12-shape entry with `agent="search_agent"`, `tool="search_fuel_news"`, `tool_input={query}`, `tool_output={summary, sources_count, fetched_at}`, `status="ok"` (or `warn` on Tavily failure per D-12). Renders via existing `TraceStep` using `AGENT_LABEL["search_agent"] = "Search agent"`. |
| **Parallel fan-out trace render (D-01)** | When `fuel_agent` and `route_agent` arrive in the same superstep, the SSE order is non-deterministic — both list items append in the order received. Their `timestamp` fields will differ by ≤ ~50ms (the demo evidence). The TracePanel does NOT visually group them (no "parallel" badge, no indentation, no connecting line) — adjacency in the list + adjacent timestamps in the expanded detail are sufficient. |
| **Feedback POST (D-16)** | `FeedbackButtons.vote()` calls `api.postFeedback({thread_id, message_id, score: 1\|-1, reason})` (the `score` is mapped 'up'→1, 'down'→-1 to match the Langfuse Score API value field per RESEARCH §Pattern 3). On 200: button stays `aria-pressed=true`, no toast (matches Phase 4 D-17 silent-confirmation). On non-200 / network error: button still shows voted, `console.error` logged, no UI surface. **A failed feedback POST never blocks any other UI flow.** |
| **Feedback localStorage mirror (optional)** | Implementer's choice whether to keep the Phase 4 localStorage mirror as a best-effort offline cache. Recommendation: **drop it** — the v1.0 UX target is "vote sent to Langfuse," not "vote stored offline." Removing the mirror simplifies the swap and matches D-16's "synchronous, fire-and-forget on the wire." |
| **Resume turn trace stream (D-08 preserved)** | A resume `POST /api/chat` with `{approve: true\|false}` produces its OWN fresh trace stream — the trace panel clears and re-populates with the post-approval entries (response_node + any cleanup). This preserves the Phase 4 D-08 invariant (trace panel = current-turn only). The pre-pause `hitl_gate` warn entry is OVERWRITTEN by the resume turn's trace, which is acceptable: Langfuse holds the full audit trail. |
