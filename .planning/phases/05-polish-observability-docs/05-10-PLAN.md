<objective>
Fix gap-3 from 05-UAT.md test 6: news/trend queries (e.g. "Why are diesel prices rising this week?") trigger the planner to re-route to `search_context` 5 times in a row until the loop-budget guard exhausts. The Tavily TTL cache prevents real API duplication, but the LLM planner is invoked 5 redundant times AND the final response prose mis-renders "I need a bit more information to calculate your surcharge. (planner_loop_budget_exhausted)" — completely off-topic for what is clearly a news query (the Market context blockquote with sources renders correctly above this misleading prose).

Two underlying issues per UAT remediation_hint:
1. **Planner loop:** `planner_node` does not check `state.search_context is not None` before re-routing. After `search_agent` populates `search_context` and returns to planner per Phase 5 D-03 (`g.add_edge("search_agent", "planner")`), the planner re-runs the LLM, which AGAIN classifies the user message as out_of_scope/news → re-emits `next_step="search_context"` → router re-dispatches to search_agent → infinite loop until D-04 budget guard fires at 5 iterations.
2. **Response prose:** `response_node._render_prose_clarify` is called whenever `surcharge_result` is None and `clarification_reason` is set. The `planner_loop_budget_exhausted` reason from D-04 sets clarification_reason; status falls into 'clarify' bucket; clarify prose renders. There is no detection of "this is a search-only flow, the search_context is the actual answer."

Plus a third quality improvement: add a dedicated `news_query` enum value so news queries don't share the `out_of_scope` bucket with truly-unsupported queries (the early-return guard covers BOTH values for backward compatibility, but the prompt update encourages the LLM to use the more specific intent).

Purpose: restore Phase 5 ROADMAP success criterion #1 (parallel and observability demos depend on responsive agent — 5 redundant planner LLM calls per news query is observable in Langfuse and undermines the Agent Architecture rubric) AND TOOL-05 acceptance criterion (search agent flow ends in a sensible response, not a loop-exhausted clarify). Restores ORCH-01 (planner correctness) and TOOL-05 user-facing quality.

- Updated `backend/agent/prompts/planner.py` adding `news_query` to user_intent enum and routing guidance
