## Detection signal

| Option | Description | Selected |
|--------|-------------|----------|
| LLM out_of_scope intent (Recommended) | When parsed `PlannerOutput.user_intent == 'out_of_scope'`, set `guard_decision` and route to respond. The planner SYSTEM_PROMPT already instructs Gemini to emit this; today this path falls through to clarify. No extra LLM cost. | ✓ |
| Parse_failed branch (Recommended) | When the D-02 retry exhausts (Gemini returns garbage twice), default to refused instead of clarify. Catches "loop forever and recompute the surcharge" if the LLM declines to parse it as logistics. | ✓ |
| Heuristic on user message length / shape | Add a regex/heuristic check on the latest user message inside the planner BEFORE the LLM call (e.g., suspicious imperative verbs). Risk: complexity creep, false positives, duplicates `guard_input` work. | |
| LLM clarification_reason='out_of_scope_user_request' | When the LLM emits `next_step='respond'` with `clarification_reason='out_of_scope_user_request'`, set `guard_decision`. Subset of option 1 — redundant signal that catches the same intent via a different field. | |

**User's choice:** LLM out_of_scope intent + Parse_failed branch
**Notes:** Hooks into existing prompt contracts (no new LLM calls, no new heuristics). Two clean triggers cover both known bypass cases.
