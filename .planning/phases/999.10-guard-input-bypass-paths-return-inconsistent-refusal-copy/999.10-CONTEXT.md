# Phase 999.10: Unify refusal copy on guard_input bypass paths - Context

**Gathered:** 2026-05-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Make the user-visible refusal experience consistent for off-topic inputs that bypass `guard_input` and reach the planner. Today, real refusals (injection / off_topic regex hits) render the locked `REFUSAL_COPY` with `status='refused'`, while bypass-but-off-topic inputs (e.g., "weather in Bangkok", "loop forever and recompute the surcharge until it equals 50%") fall through the planner's `parse_failed` or LLM-emitted `out_of_scope` paths and render the generic `planner_parse_failed` clarify copy with `status='clarify'`.

This phase unifies the planner-side refusal paths onto the same `REFUSAL_COPY` + `status='refused'` contract as `guard_input` refusals, by setting `state.guard_decision` from inside `planner_node` and reusing the existing `response_node` refusal branch.

In scope: changes inside `planner_node` and (potentially) the `guard_decision` schema/category vocabulary. Out of scope: tightening `guard_input`'s `_DOMAIN_ALLOW_PATTERNS`, adding new guard nodes, adding cost-bombing intent regexes, and any FE `FinalStatus` Literal extension.

</domain>

<decisions>
## Implementation Decisions

### Fix location
- **D-01:** `planner_node` sets `state.guard_decision = {layer: 'input', refused: True, category: <new>, violations: []}` and returns `next_step='respond'` on the qualifying branches. Reuses the existing `response_node` refusal branch verbatim — zero new graph wiring, zero new edges, smallest blast radius.
- **D-02:** No new node is inserted between `planner` and `response`. No new edges. `recursion_limit=12` budget unchanged.
- **D-03:** `_DOMAIN_ALLOW_PATTERNS` in `backend/agent/nodes/guard_input.py` is NOT modified by this phase. The "weather in Bangkok" admit-by-allow-list behavior is accepted upstream; the planner becomes the second-line defense.

### Detection signal
- **D-04:** Trigger 1 — when parsed `PlannerOutput.user_intent == 'out_of_scope'`, set `guard_decision` (category `'planner_off_topic'`) and route to respond INSTEAD of falling through to clarify. The planner SYSTEM_PROMPT already instructs Gemini to emit this for non-logistics requests; this hooks into an existing prompt contract rather than adding a new one.
- **D-05:** Trigger 2 — when the D-02 retry loop exhausts (`planner parse attempt 2 failed`), set `guard_decision` (category `'planner_parse_failed'`) and route to respond INSTEAD of returning `next_step='clarify', clarification_reason='planner_parse_failed'`. Refusal is unconditional on this branch — no further conditioning on user message content.
- **D-06:** No new heuristic regex on the latest user message inside the planner. No duplicate-of-`guard_input` work in the planner.
- **D-07:** No second-pass classifier (rules or LLM) added inside the parse_failed branch. The trigger is the parse exhaustion itself.

### Status / observability
- **D-08:** Wire `status='refused'` is preserved across BOTH guard-tripped (existing) and planner-tripped (new) refusal paths. D-03 from quick task 260509-utd remains in force: judges and adversarial classmates see the same locked `REFUSAL_COPY` line on every refusal regardless of which layer caught it.
- **D-09:** Differentiation lives entirely in `state.guard_decision.category`. New category vocabulary (extending the `GuardCategory` Literal in `backend/agent/nodes/guard_input.py`):
  - existing: `allow | injection | off_topic | cost_bombing | unclear`
  - additions for this phase: `planner_off_topic | planner_parse_failed`
- **D-10:** `state.guard_decision.layer` stays `'input'` for both the existing `guard_input` trips AND the new planner trips (matches `response_node`'s `status = 'refused' if layer == 'input' else 'guard_failed'` predicate at `backend/agent/nodes/response_node.py:201-202`). The `category` field is the differentiator, not the layer.
- **D-11:** Trace entry shape on the new path: emitted by `response_node` (NOT by `planner_node`), tagged `agent='response'`, mirroring the existing guard refusal branch behavior. The planner does NOT emit a separate refusal trace entry in addition to its normal planner trace — the planner's own trace entry can stay as today (the LLM call happened, it produced a parsed/unparsed output) and the response node's refusal trace records the verdict.
- **D-12:** FE `FinalStatus` Literal extension is OUT OF SCOPE — `'refused'` is already in the FE FinalStatus vocabulary (added by quick task 260509-utd). No FE work required by this phase.

### Scope
- **D-13:** Targeted planner-side only. The two known bypass cases (weather/Bangkok, loop-forever-recompute-surcharge) are addressed by D-04 and D-05 respectively. The "weather in Bangkok" path relies on Gemini correctly tagging `user_intent='out_of_scope'` per the existing planner SYSTEM_PROMPT contract — if Gemini does not, the parse_failed branch (D-05) is the second-line catch.
- **D-14:** No cost-bombing intent regex added. "Loop forever" / "recompute until X" linguistic patterns are NOT special-cased — they are caught either by the LLM tagging `out_of_scope` or by parse_failed exhaustion.
- **D-15:** No interaction with phase 999.11 (SSE hang investigation). The two issues share a discovery context (260509-utd live probe) but distinct root causes; do not conflate.

### Claude's Discretion
- Exact regex/code shape of the planner refusal helper function (a small `_set_guard_refusal(category)` helper or inline dict construction is fine).
- Whether to refactor the existing planner parse_failed return into the new helper or keep it inline.
- Test fixture organisation: extend `test_planner.py` with the two new categories, OR add a focused `test_planner_refusal_paths.py` module — the planner judges either way.
- Whether to log a structured observability hint (e.g., `logger.info("planner triggered refusal: category=%s", category)`) on the new branches; not required by D-11 but harmless.
- Whether `_loop_budget_exhausted` should be considered another refusal trigger (currently it returns `next_step='respond'` with `clarification_reason='planner_loop_budget_exhausted'` — same clarify-style fall-through as parse_failed). Phase research / planning may surface this as an extension or leave it as-is.

</decisions>

## Existing Code Insights

### Reusable Assets
- `state.guard_decision` field already exists on `AgentState` (added by 260509-utd) — no schema change.
- `response_node`'s guard refusal branch (`response_node.py:199-235`) already keys off `state.guard_decision.refused` and renders `REFUSAL_COPY` with `status='refused'` (when `layer='input'`) — no change needed.
- `REFUSAL_COPY` constant in `backend/agent/prompts/guard.py` is the locked string — reused verbatim.
- `GuardCategory` Literal in `backend/agent/nodes/guard_input.py:47` is the extension point for `'planner_off_topic'` and `'planner_parse_failed'` — single Literal expansion.
- D-02 retry loop (`planner.py:213-236`) already has the parse_failed branch as a single early-return point — clean place to swap clarify-return for guard_decision-return.
- LLM out_of_scope intent path goes through the post-LLM merge today and falls through to clarify — clean place to insert the early-return that sets `guard_decision`.

### Established Patterns
- Pattern from 260509-utd: refusal verdict is a dict, not a sentinel. `{layer, category, refused, violations}` shape. Re-use exactly.
- Pattern from `_loop_budget_exhausted`: planner can return `{"next_step": "respond", ...}` with no Gemini call and no full extraction-merge. The new D-04 / D-05 branches follow the same shape but additionally set `guard_decision`.
- Pattern from existing `response_node` refusal branch: appends assistant message into `state.messages` for replay parity. Already correct — no changes needed for the new categories because `response_node` keys on `refused` boolean, not on category.
- Pattern from `_route_from_guard_input`: conditional-edge routing based on `guard_decision.refused`. Not needed for this phase because the planner's `next_step='respond'` already routes correctly.

### Integration Points
- `planner.py` ↔ `response_node.py`: contract via `state.guard_decision` + `next_step='respond'`. No new contract; uses the existing one in a new caller.
- `state.py`: NO schema change to `AgentState`.
- `guard_input.py`: `GuardCategory` Literal extension only (additive). No node logic changes.
- FE: zero changes. `FinalStatus = 'refused'` already accepted; the new `guard_decision.category` values flow through the existing `final_payload.status` path opaquely.
- Langfuse traces: new category strings appear in the existing `tool_output.guard_category` field on the response refusal trace — Langfuse picks them up automatically; no SDK or callback changes.

<specifics>
## Specific Ideas

- The behavior gap was discovered against a 5-case live probe inside `quick-260509-utd`: cases 1 (injection) and 3 (recipe) returned `status='refused'` + `REFUSAL_COPY`; cases 2 (weather/Bangkok) and 4 (loop forever) returned `status='clarify'` with the parse_failed copy. After this phase, all four should look identical to a judge: `REFUSAL_COPY` + reasoning trace showing a refusal.
- The user's framing is consistent: "judges should see the same line on every refusal" (D-03 invariant from 260509-utd) — preserved.
- Preferred approach is to hook into existing prompt contracts (the planner already instructs Gemini to emit `out_of_scope`) rather than add new heuristics or LLM calls.
- The user explicitly rejected `_DOMAIN_ALLOW_PATTERNS` tightening because it could create false refusals (e.g., "is Bangkok traffic bad today" would be at risk if 'bangkok' alone were dropped from the allow-list).

</specifics>

<deferred>
## Deferred Ideas

- **Tighten `_DOMAIN_ALLOW_PATTERNS`** — would require auditing the demo question bank for legit single-token "Bangkok" queries. Defer to a separate hardening phase if dry-run shows residual misses after this phase ships.
- **Cost-bombing intent regex in `guard_input`** — separate detection for "loop forever" / "recompute until X" linguistic patterns. Cleaner separation from off_topic, but would belong in a `guard_input`-side phase, not this planner-side phase.
- **`_loop_budget_exhausted` as a third refusal trigger** — currently returns clarify with `clarification_reason='planner_loop_budget_exhausted'`. Could be folded into `'planner_parse_failed'` or its own category. Planning may surface this; leave as-is unless research / planning identifies a concrete user-facing inconsistency.
- **FE `FinalStatus` Literal differentiation per category** — e.g., a "(off-topic)" or "(could not understand)" sub-tag in the FE refusal banner. Out of scope; FE renders the same `REFUSAL_COPY` markdown today.
- **Langfuse Score auto-attachment for the new refusal categories** — RESEARCH OQ from 260509-utd suggested a `guard_trip` Score reusing the per-turn `seed_trace_id` plumbing. Same deferral reasoning as 260509-utd.
- **Phase 999.11 (SSE hang investigation)** — distinct root cause; not coupled.

</deferred>
