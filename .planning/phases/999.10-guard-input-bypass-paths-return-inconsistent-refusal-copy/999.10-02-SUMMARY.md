## Decisions Made

- **Helper over inline dicts** — `_set_guard_refusal(category)` keeps the verdict-dict invariants (layer='input', refused=True, violations=[]) in exactly one place. Both call sites stay declarative ("refuse with this category, route to respond"). CONTEXT.md left this as Claude's Discretion; DRY won.
- **D-04 placement before origin_hub_id validation + 999.1 merge** — out_of_scope short-circuits BEFORE any logistics-extraction machinery runs, because that machinery's semantics (hub lookup, follow-up null-out, fan-out promotion) do not apply to refused messages. Keeps the refusal return shape minimal: just next_step + guard_decision, no shipping_type / weight_kg / origin / destination / origin_hub_id / missing_fields / clarification_reason / reasoning_trace.
- **D-05 unconditional refusal** — no further conditioning on user message content (D-06/D-07 invariant). The trigger is the parse exhaustion itself; adding a second-pass classifier would duplicate guard_input's work and risk false-clarifies on legit messages that happen to crash JSON parsing.
- **D-11 trace ownership stays with response_node** — planner emits NO refusal trace entry of its own. The existing `response_node` refusal branch already emits an `agent='response'` trace entry; adding a planner-tagged refusal trace would either duplicate that entry (poor signal) or split the refusal across two trace nodes (split observability). Keeping it single-source matches the existing guard_input -> response_node refusal flow.
- **logger.info lines on both branches** — Claude's Discretion item from CONTEXT.md. Non-load-bearing for behaviour or tests; load-bearing for Langfuse trace-side observability comparison during the live adversarial-pack rerun that closes the milestone. One INFO line per branch, no DEBUG noise.
