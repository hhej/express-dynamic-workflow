## Resolution

root_cause: NOT DEFINITIVELY REPRODUCED IN UNIT TESTS. Backend was exhaustively audited and ruled out as a source of duplicate rows (`_attach_message_ids` and `response_node` are structurally incapable of producing duplicate assistant entries). The remaining surface area is a frontend reconciliation/effect-firing race in the `done` `useEffect` + `handleResume` interaction in ChatApp.tsx — but no unit-test scenario (StrictMode + MSW + fake SSE) ever managed to fire the duplicate, before OR after the applied defensive changes. The most credible mechanism remains: a render path that re-evaluated the `done` effect post-resume with stale `chat.finalPayload` still visible (or with `lastAppendedPayloadRef.current === null` against a still-truthy finalPayload), causing a re-append onto the freshly-replayed messages. Whether the dep-narrowing or the guard-seed is load-bearing — versus both together — versus the prior-shipped band-aid having coincidentally covered something else — cannot be distinguished from the evidence. Manual end-to-end verification confirms the symptom is gone, so the cumulative diff is treated as the resolution.

fix:
  - frontend/components/ChatApp.tsx: narrowed `done` useEffect deps from `[chat.finalPayload, chat.status, conversations.refresh]` to `[chat.finalPayload, chat.status]`. Behaviourally a no-op today (`conversations.refresh` is already memoized with `useCallback([], [])`), but removes a future-regression foot-gun if `refresh`'s identity ever destabilises. May or may not have been load-bearing for the user-reported bug.
  - frontend/components/ChatApp.tsx: in `handleResume`, seed `lastAppendedPayloadRef.current = chat.finalPayload` BEFORE `chat.setThreadId(threadId)` instead of clearing it to `null` after. This arms the ref-equality short-circuit against the prior turn's already-appended payload identity, so any render path that re-evaluates the `done` effect with the OLD finalPayload still in scope would short-circuit at the guard rather than re-append. May or may not have been load-bearing.
  - frontend/components/chat/MessageList.tsx: reverted the assistant `<li>` key from `a-${m.id}-${i}` (band-aid from quick task 260509-e0p) back to `a-${m.id}`. The band-aid was masking, not fixing — it suppressed the duplicate-key React warning while leaving the duplicate logical row in the messages array. Removing it restores future reconciliation-bug signal.
  - frontend/__tests__/components/ChatApp.duplicate-resume.repro.test.tsx: NEW reproduction test covering Scenarios A (send → done → resume own), B (send → done → resume other → resume own), C (send → done → resume own → follow-up send). All three pass both before and after the applied changes — meaning the test path never reproduced the user's bug. Kept as a regression net for any future change that *would* introduce a duplicate via these three flows.
