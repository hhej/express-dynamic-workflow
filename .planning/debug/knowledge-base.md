## fix-eppo-scraper-url-restructure — EPPO petroleum-statistics URL slug renamed and Excel structure changed; scraper failed with 404
- **Date:** 2026-05-09
- **Error patterns:** requests.exceptions.HTTPError, 404 Client Error, Component not found, en-petroleum-statistic, _scrape_eppo, fetch_fuel_prices, eppo.go.th, dashboard 7d empty, dashboard 30d empty, fetched 0 rows from EPPO
- **Root cause:** EPPO renamed the Joomla menu slug from `en-petroleum-statistic` to `petroleumprice-statistic` and restructured the page to expose pivoted Excel tables (P03..P09). The original `_scrape_eppo()` both targeted the dead URL and assumed a flat-header CSV/Excel — it could not parse P09's pivoted MIN/WT.AVG/MAX-by-month layout. The legacy `xlrd` driver was also missing from the venv, blocking BIFF .xls reads. P09 alone is monthly; the published daily snapshot lives on a separate oil-share PHP page.
- **Fix:** Rewrote `_scrape_eppo()` to (1) download `P09.xls` from the new direct URL, (2) parse via a dedicated `_parse_p09_workbook()` that walks the "2003-current" sheet and emits one row per month from the WT.AVG bucket of column 8 (HSD B7), (3) optionally pull today's daily snapshot via `_scrape_oil_share_today()` (parses `oil_name6v2.png` row label on the EPPO oil-share page), and (4) merge with the seed CSV (append only rows strictly newer than the seed max, dedup on date). Added `xlrd==2.0.2` to requirements. Added `TestScrapeEppoInternals` (5 tests) for the new internals. Public contract preserved.
- **Files changed:** data/scripts/fetch_fuel_prices.py, data/raw/eppo_diesel_prices.csv, requirements.txt, backend/tests/test_fuel_prices.py

## 999.5-fix-resume-flow-appending-duplicate-assistant-message — Resume-flow duplicate assistant row + duplicate-key React warning, masked by an array-index key band-aid
- **Date:** 2026-05-09
- **Error patterns:** Encountered two children with the same key, a-{thread_id}-{turn_idx}, MessageList.tsx, ChatApp.tsx, handleResume, lastAppendedPayloadRef, done useEffect, conversations.refresh, duplicate assistant bubble, doubled assistant row, resume from sidebar
- **Root cause:** Not definitively reproduced in unit tests. Backend ruled out exhaustively (`_attach_message_ids` + `response_node` are structurally incapable of duplicating assistant rows). Most-credible mechanism is a render path in ChatApp.tsx that re-evaluated the `done` `useEffect` post-`handleResume` with stale `chat.finalPayload` still visible (or with `lastAppendedPayloadRef.current === null` against a still-truthy finalPayload), re-appending the prior turn's payload onto the freshly-replayed messages. A prior quick task (260509-e0p) had shipped a React-key band-aid (`a-${m.id}-${i}`) that suppressed the warning without removing the duplicate logical row.
- **Fix:** (1) Narrowed `done` useEffect deps in ChatApp.tsx from `[chat.finalPayload, chat.status, conversations.refresh]` to `[chat.finalPayload, chat.status]` — `refresh` is stable today via `useCallback([], [])` but the dep added a future-regression foot-gun. (2) In `handleResume`, seeded `lastAppendedPayloadRef.current = chat.finalPayload` BEFORE `chat.setThreadId()` instead of clearing it to `null` after, so any race re-firing the `done` effect with stale finalPayload short-circuits at the guard. (3) Reverted the MessageList.tsx assistant key from `a-${m.id}-${i}` back to `a-${m.id}` — the suffix was masking, not fixing. (4) Added a regression test covering Scenarios A/B/C (resume-own, resume-other-then-back, resume-then-followup-send) — passes both before and after the diff, kept as a future-regression net. Manual user repro confirmed the symptom is gone.
- **Files changed:** frontend/components/ChatApp.tsx, frontend/components/chat/MessageList.tsx, frontend/__tests__/components/ChatApp.duplicate-resume.repro.test.tsx
- **Lesson:** A React key suffixed with the array index is a band-aid that hides reconciliation bugs — remove it and audit the source of duplicate ids instead of suppressing the warning. When unit tests can't reproduce a user-reported reconciliation bug despite faithful step-replay, treat manual end-to-end repro as the authoritative verification signal — but be honest in the resolution about what was confirmed vs. inferred.
